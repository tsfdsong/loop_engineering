# A+D 引擎：验收条件推理 + 自信度闸门

A+D 引擎是 loop 默认模式的核心——自动将模糊需求转化为可验证的 BDD 示例，用自信度决定交互深度。

## A: 验收条件自动推理

### 输入
- 用户功能描述
- 项目上下文（技术栈、已有模块、架构约束）

### 输出
```
验收条件清单:
1. [G4] 积分累积 API 正确响应（单元测试+集成冒烟）
2. [G4+G6] 积分字段使用 decimal(10,2)（前后端一致）
3. [G4+G5] 积分明细查询响应 < 200ms
4. [F1+F4] 前端页面控制台 0 错误，交互流程正确
```

每个验收条件必须绑定到具体的门禁维度。不能绑定 = 标记为"模糊条件"。

### BDD 示例生成

对每条验收条件生成至少 1 个 BDD 示例：

```
验收条件 1: 积分累积 API 正确

场景: 正常累积
  Given 用户 A 当前有 100 积分
  When  触发累积事件 (+50.5 积分)
  Then  余额 = 150.5
        AND 积分明细表新增 1 条记录

场景: 边界值  
  Given 用户 B 当前有 0 积分
  When  触发累积事件 (+0.01 积分)
  Then  余额 = 0.01
```

## D: 自信度闸门

对每个验收维度评估自信度，决定交互方式：

### 自信度判定（纯规则，零 token）

```
🟢 高自信度 → 审计闸门（自动通过）:
  - 从项目文件直接提取的信息（技术栈、架构约束）
  - 用户已明确指定的条件
  - 有明确约束规则可套用的

🟡 中自信度 → 软闸门（AskUserQuestion, 5s 倒计时自动采纳推荐）:
  - 需要做"选择"的（扩展现有 vs 新建）
  - 有多种合理方案的
  - 涉及 trade-off 的

🔴 低自信度 → 深度闸门（生成 BDD 示例验证）:
  - 依赖外部服务的（"对接 XX 支付网关"）
  - 涉及新技术/新模式的
  - 需求描述中有模糊词的（"用户体验要好"）
  - 影响核心链路的（"改用户认证逻辑"）
```

### 深度闸门行为

触发 🔴 时，不弹出确认框，而是自动生成 BDD 示例：

```
📋 积分兑换规则 — 我推理了 3 种理解:

  方案A【推荐】: 固定积分兑换固定礼品
    Example: 用户100积分 → 兑换"50元优惠券" → 扣100积分

  方案B: 动态积分兑换
    Example: 用户100积分(系数0.1) → 兑换10元商品

  方案C: 阶梯兑换
    Example: 0-100积分 1:1, 100以上 1:0.8

AskUserQuestion:
  问题: "采用哪种兑换方案？"

  ┌─────────────────────────────────────────────────┐
  │ 【推荐】方案A：固定积分兑换                       │
  │   实现最简单，与现有 points 表结构直接兼容         │
  │ 【备选】方案B：动态积分兑换                       │
  │   不推荐：增加兑换系数配置表，引入额外复杂度       │
  │ 【备选】方案C：阶梯兑换                           │
  │   不推荐：阶梯规则需额外配置，当前需求未明确需要   │
  └─────────────────────────────────────────────────┘
```

## 验收条件→门禁维度映射

| 验收条件类型 | 默认门禁维度 |
|-------------|-------------|
| 功能正确性 | G4 + G5 |
| 性能要求 | G4(性能测试) + G5(真实压测) |
| 前端交互 | F1 + F3 + F4 |
| 数据类型一致 | G4 + G6 |
| 构建/部署 | G7 + G8 |
| 环境连通 | G1 |

如果某个验收条件无法绑定到任何门禁维度 → 标记为"模糊条件" → 必须触发 🔴 深度闸门让用户澄清。
