# System Review — 三步系统级审查方法论

从"审查项目"层面的三类常见问题出发，串联三条方法论。每步末尾括号内附 ATAM/arc42 业界参考检查点。

---

## 核心定位

| 维度 | system-review（本技能） | code-reviewer | production-code-audit |
|------|------------------------|----------------|------------------------|
| 审查对象 | 项目/模块/系统 | 单文件/单 PR | 全量代码逐行 |
| 关注重点 | 一致性、职责、改进 | 代码质量、安全 | 安全/性能/生产就绪 |
| 典型触发词 | "审查项目""架构有没有问题""整体一致性" | "code review" | "生产就绪" |
| 适用阶段 | 设计/重构/集成/重大变更 | 每次 PR | 上线前/定期审计 |

> **三技能不互斥，是不同粒度的审查。** 一次大型重构可能同时用三个。

---

## 三步方法论总览

```
Step 1: 横向自洽性检查 ── 组件间是否一致
    ↓ (找出矛盾点)
Step 2: 纵向架构深度审查 ── 职责是否各司其职
    ↓ (评估架构质量)
Step 3: 持续改进审查 ── 哪些可增强/精简/优化
    ↓ (输出改进路线图)
```

每步都是**独立可用**的，但串联使用最系统。

---

## Step 1: 横向自洽性检查

**核心问题**：系统的各个组件之间是否存在矛盾——接口契约对不上、字段命名不一致、假设互相冲突。

### 检查清单

| 类别 | 检查项 | 参考 |
|------|--------|------|
| **接口契约** | 模块A声称的输入/输出，与模块B实际调用的参数类型/命名是否一致？ | (ATAM: 场景模块 / arc42: §3 Context) |
| **字段对齐** | 同一概念在不同层（如 API/DTO/DB）的字段名/类型/可选性是否一致？ | (arc42: §5 Building Block) |
| **命名一致** | 核心实体（如 `User`）的命名在所有引用点是否统一？ | (arc42: §12 Glossary) |
| **配置假设** | 各模块对环境/配置/权限的假设是否互相矛盾？ | (ATAM: 质量属性 / arc42: §2 Constraints) |
| **错误处理约定** | 各模块的错误码/异常类型/重试策略是否一致？ | (arc42: §6 Runtime View) |
| **状态机一致** | 跨模块的同一对象状态定义是否相同？ | (ATAM: 场景) |
| **数据流向** | 数据从输入到持久化的路径上，各环节对数据结构的假设是否一致？ | (arc42: §5 Building Block) |
| **协议版本** | 客户端/服务端的 API 版本、序列化格式是否对齐？ | (ATAM: 性能/可用性) |

### 检查步骤

1. 列出系统的核心组件（如 API 层、领域层、基础设施层、外部服务）
2. 对每个**跨组件边界**，检查上表所有检查项
3. 记录每个矛盾点：**预期 vs 实际**，影响范围
4. 按影响范围排序：阻塞型（运行时会崩）> 隐藏型（边缘场景）> 优雅降级型

### 输出格式

```
## 自洽性检查报告
- [CRITICAL] API 层的 User.id 类型是 string，DB 层是 int → 类型不一致
- [HIDDEN] 订单状态的"已取消"在模块A是 CANCELLED，在模块B是 CANCELED
- [MINOR] 日志格式约定在用户服务是 JSON，在订单服务是 key=value
```

## Step 2: 纵向架构深度审查

**核心问题**：系统的分层、职责、链路是否合理？每个组件是否做且只做它该做的事？

### 检查清单

| 类别 | 检查项 | 参考 |
|------|--------|------|
| **职责单一** | 每个模块/类/函数是否只做一件事？是否存在"上帝类"或"上帝模块"？ | (ATAM: 可修改性 / arc42: §5 Building Block) |
| **分层清晰** | 依赖方向是否单向（如 UI→Service→Domain→Infra）？是否有反向依赖？ | (ATAM: 性能/可维护性 / arc42: §4 Solution Strategy) |
| **链路完整** | 一个核心用例（如"下单"）的端到端链路是否能画出来？有没有断点？ | (ATAM: 场景 / arc42: §6 Runtime View) |
| **抽象恰当** | 抽象层级是否一致？有没有"上帝接口"或"过度抽象"？ | (ATAM: 可修改性) |
| **质量属性** | 性能/可用性/安全/可修改性在架构中是否被显式处理？ | (ATAM 核心) |
| **决策可追溯** | 重要架构决策是否记录了理由（ADR）？ | (arc42: §9 Architectural Decisions) |
| **风险可见** | 已知技术债和风险是否被记录和追踪？ | (arc42: §11 Risks) |
| **部署合理** | 架构是否与部署模型一致（微服务/单体/无服务器）？ | (arc42: §7 Deployment View) |

### 检查步骤

1. 画出系统的**组件图**（按层次而非按功能）
2. 对每个组件检查"职责单一"和"抽象恰当"
3. 对每条**跨层调用**检查依赖方向
4. 选 3-5 个**核心用例**，逐个追踪端到端链路
5. 对每条链路检查质量属性是否被处理
6. 检查 ADR 和技术债记录

### 输出格式

```
## 架构深度审查报告
- 职责: ⚠️ OrderService 同时处理订单逻辑、支付调用、消息发送 → 拆分
- 分层: ✅ 依赖方向单向，无反向
- 链路: ❌ "取消订单"用例在 OrderService 后断点，库存回滚缺失
- 质量: ⚠️ 可修改性低，OrderService 修改常触发 PaymentService 改动
- 决策: ❌ 5 个重要决策无 ADR
```

## Step 3: 持续改进审查

**核心问题**：当前系统中哪些可以增强、精简或优化？

### 检查清单

| 类别 | 检查项 | 参考 |
|------|--------|------|
| **冗余代码** | 是否存在功能重复的代码/类/模块？是否能合并？ | (arc42: §5 Building Block) |
| **过时文档** | 文档/SKILL.md/注释是否与实际代码一致？ | (ATAM: 可维护性) |
| **废弃功能** | 是否有已标记废弃但仍在使用/导出的功能？ | (ATAM: 可修改性) |
| **可强化点** | 哪里缺少错误处理？缺少测试？缺少日志？缺少验证？ | (ATAM: 可用性/安全) |
| **简化机会** | 是否有过度设计、过度抽象、可以简化的部分？ | (arc42: §4 Solution Strategy) |
| **性能瓶颈** | 是否有明显的 N+1、重复计算、不必要的 IO？ | (ATAM: 性能) |
| **安全加固** | 是否所有输入都验证？敏感操作都审计？ | (ATAM: 安全) |
| **可观测性** | 关键路径是否有日志/指标/追踪？ | (ATAM: 可用性 / arc42: §6) |

### 检查步骤

1. 从 Step 1/Step 2 的报告出发，收集**改进候选**
2. 按"投入产出比"排序：高价值低投入 > 低价值高投入
3. 区分**立即做**（安全/阻塞）、**下迭代做**（性能/增强）、**技术债登记**（非紧急但需追踪）
4. 写入项目级 TODO 或 issue 跟踪

### 输出格式

```
## 改进路线图

### 🔴 立即做（安全/阻塞）
- [ ] 修复 OrderService 职责拆分（影响所有订单相关功能）
- [ ] 补全"取消订单"库存回滚（当前存在数据不一致风险）

### 🟡 下迭代做（增强）
- [ ] 为核心服务添加结构化日志
- [ ] 缓存热点数据（订单详情、商品信息）
- [ ] 补充 5 个核心场景的集成测试

### 🟢 技术债登记
- [ ] 5 个架构决策缺少 ADR
- [ ] 用户服务的密码加密逻辑需要迁移到统一 crypto 模块
```

## 执行模式

### 快速模式（15-30 分钟）

适用于：日常小变更后、项目交付前

```
Step 1: 自洽性检查 → 只检查接口契约 + 字段对齐（约 10 个检查点）
Step 3: 改进审查 → 只记录立即要做的（不超过 5 条）
```

### 完整模式（1-3 小时）

适用于：重大重构前、新模块集成、季度审查

```
完整执行三步，每步的完整检查清单都过一遍
```

### 深度模式（1-2 天）

适用于：年度架构审查、并购评估

```
三步 + ATAM 质量属性场景 + arc42 12 节全面文档
由多个 ZCode session 并行执行不同组件的审查
```

## 局限

- **不替代代码级审查** — 代码异味、bug、安全漏洞走 code-reviewer 或 production-code-audit
- **不替代实际测试** — 改进路线图中的"补全测试"需要 testing-patterns 配合
- **不替代人工判断** — system-review 的输出是 ZCode 的分析，最终决策仍需用户确认
- **不适用于紧急 hotfix** — 紧急修复场景走 systematic-debugging

## 附录 · 设计哲学背靠（Ousterhout · A Philosophy of Software Design）

> **v6.4 新增** — Ousterhout《A Philosophy of Software Design》核心思想作为 system-review Step 2 纵向架构深度审查的**理论背靠**。原 code-engineering 技能拆分时将本节内容迁入 system-review 技能，让"系统级审查"与"系统级设计哲学"在同一处可查。

### A.1 复杂度是渐增的

> 核心论断：**软件复杂度不是单一决策的产物，而是无数小决策的累积。**

- 每次"为现在省事"的设计选择都会增加未来的认知负担
- 复杂度 = 依赖（模块间耦合）+ 晦涩（代码意图不清晰）+ 重复
- **战术编程 vs 战略编程**：
  - 战术编程：写完能跑就完（短期快，长期难）
  - 战略编程：每次决策考虑未来 6 个月的可维护性

**system-review 应用**：
- Step 3 持续改进审查时，问每个改动"是战术还是战略？"
- "战术编程的积累"本身就是技术债

### A.2 深模块 vs 浅模块

> 核心论断：**少量复杂接口（深模块）> 大量简单接口（浅模块）。**

```
浅模块（差）：
┌────────────────────┐
│ getName()          │
│ getEmail()         │
│ getAge()           │
│ getAddress()       │ ← 10 个简单 getter
│ getPhone()         │
│ ...                │
└────────────────────┘
接口复杂，实现简单 → 调用方累

深模块（好）：
┌────────────────────┐
│ sendInvitation()   │ ← 1 个复杂接口
└────────────────────┘
接口简单，实现可以复杂 → 调用方轻松
```

**深模块特征**：
- 强大功能 / 简单接口
- 隐藏实现细节
- 调用方不需要理解内部

**system-review 应用**：
- Step 2 抽象恰当检查：每个模块是"深"还是"浅"？
- 浅模块是"重新发明参数"的反模式

### A.3 信息隐藏（Information Hiding）

> 核心论断：**模块不应暴露不必要的信息。**

| 应隐藏 | 不应暴露 |
|-------|---------|
| 内部数据结构 | 数据库 schema 细节 |
| 算法选择 | 配置格式 |
| 性能优化细节 | 内部状态 |
| 第三方库封装 | 异常实现细节 |

**system-review 应用**：
- Step 2 抽象恰当：模块边界是否清晰？
- "穿层调用"（UI 直接查 DB）= 信息隐藏失败

### A.4 复杂度来源（4 大类）

| 来源 | 描述 | system-review 检测 |
|------|------|-------------------|
| **依赖** | 模块间不必要的耦合 | 依赖图分析、循环依赖 |
| **晦涩** | 代码意图不清晰 | 命名/注释质量 |
| **重复** | 相同逻辑多处出现 | 重复率检测 |
| **复杂性累积** | 每个小决策叠加 | 战术编程积累检测 |

### A.5 设计策略

| 策略 | 实现 |
|------|------|
| **正交性** | 模块互不依赖（改 A 不影响 B） |
| **深模块** | 用小接口封装复杂实现 |
| **信息隐藏** | 不暴露不必要的信息 |
| **分层抽象** | 每层只关注自己职责 |
| **小接口、大实现** | 减少调用方负担 |

### A.6 设计哲学与 system-review 三步方法论的对应

| system-review 步骤 | Ousterhout 背靠 |
|-------------------|----------------|
| Step 1 横向自洽性 | 信息隐藏失败会引发不一致（接口穿层 → 字段不对齐） |
| Step 2 纵向架构深度 | 浅模块 / 深模块 / 正交性 / 抽象恰当 |
| Step 3 持续改进 | 战术编程 vs 战略编程 / 复杂度累积检测 |

### A.7 system-review 决策启发（Ousterhout 衍生）

1. **新增功能时**：选深模块设计，避免"快速加 5 个 getter"
2. **重构时**：问"这是把浅模块改深，还是把深模块拆浅？"
3. **审查时**：每个模块问"它的接口能更简单吗？"
4. **决策时**：用 6 个月视角判断战术 vs 战略

### A.8 不适用场景

- ❌ **过度抽象**：深模块不意味着"全部隐藏到不可见"
- ❌ **架构层次过多**：分层抽象不是"无限分"
- ❌ **回避性能**：信息隐藏不能以性能为借口

### A.9 完整内容来源

原 code-engineering 技能（v6.4 已拆分）的设计哲学要点已融入本附录。完整 Ousterhout 原书《A Philosophy of Software Design》建议阅读（特别是第 5-9 章：复杂度、深模块、信息隐藏）。

### N.1 4 维度核对清单（原 §8.3 · 触发后用 AskUserQuestion 让用户选维度）

触发核对时，**必须先用 AskUserQuestion 让用户选本次核对维度**（multiSelect: true，至少必选 1 个）：

| # | 维度 | 详细检查清单 | 方法论语源 |
|---|------|------------|-----------|
| 1 | **需求↔实现** | 最初计划/PRD 清单 vs 已实现代码，逐条打勾；发现 **[GAP]**（需求有代码无）或 **[EXTRA]**（代码有需求无）；逐条核对验收条件是否全部满足 | 本红线自含（非 system-review 三步方法论范围） |
| 2 | **模块间横向一致性** | • 接口契约（模块 A 声称的输入/输出 vs 模块 B 实际调用）<br>• 字段对齐（同一概念在 API/DTO/DB 层的命名/类型/可选性）<br>• 命名一致（核心实体在所有引用点统一）<br>• 错误处理约定（错误码/异常类型/重试策略一致）<br>• 状态机一致（跨模块同一对象状态定义相同）<br>• 协议版本（客户端/服务端 API 版本对齐） | `system-review` **Step 1 横向自洽性**（本 SKILL.md L57-88） |
| 3 | **端到端链路打通** | • 选 2-3 个核心用例，从入口追到出口<br>• 找断点（链路中缺失环节）<br>• 验证异常路径是否打通（非仅 happy path）<br>• 检查跨层数据流向是否一致 | `system-review` **Step 2 纵向架构深度 · 链路完整**（本 SKILL.md L92-127） |
| 4 | **文档↔代码一致性** | • README / API 文档 / CHANGELOG 是否与当前代码状态一致<br>• 注释是否反映当前实现<br>• 配置示例是否可运行<br>• 架构图是否反映当前分层 | `system-review` **Step 3 持续改进 · 过时文档**（本 SKILL.md L131-172） |

> **默认推荐**：首次 AskUserQuestion 选项含全部 4 维度（标注 `(推荐)`）+ 轻量版（仅 1+3）+ 单维度查。方法论不在本节重复——维度 2/3/4 的检查清单引 system-review 三步方法论对应步骤（已在表中标注本 SKILL.md 行号）。

### N.3 处置协议（原 §8.5 · 每个不一致项或分组必走 AskUserQuestion）

发现不一致 → **不得**自行决定修复/忽略，必须为每个差异项（或合理分组）调 AskUserQuestion：

| 选项 | 含义 | 推荐场景 |
|------|------|---------|
| **立即修复** | 阻断下个任务，先修这个 | CRITICAL / 阻塞型（默认 `(推荐)`） |
| **登记后续修复** | 记入 TODO，继续下个任务 | HIDDEN / 非阻塞型 |
| **标记已知边界** | 不修但文档说明 | 设计取舍 / 暂不支持 |
| **忽略** | 明确不修也不记录 | 误报 / 无意义 |

> **与 V2/C2 红线协同**：处置本身就是决策点，必须走 AskUserQuestion（与红线 3/红线 7 同级），禁止文字列选项（AGENTS.md C2 用户交互红线 + V2 验证 Gate 红线协同）。
