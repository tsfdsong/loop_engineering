# System Review — 三步系统级审查方法论

> 🔴 **用户交互红线**：遵循 skill-hub 的 4 项硬要求——必须用 `AskUserQuestion` 列出选项（含推荐），推荐项标 `(推荐)` 并说明理由，不推荐项必须说明理由，禁止自由文本输入和开放式追问。

从"审查项目"层面的三类常见问题出发，串联三条方法论。每步末尾括号内附 ATAM/arc42 业界参考检查点。

---

## 核心定位

| 维度 | system-review（本技能） | code-reviewer | production-code-audit |
|------|------------------------|----------------|------------------------|
| 审查对象 | 项目/模块/系统 | 单文件/单 PR | 全量代码逐行 |
| 关注重点 | 一致性、职责、改进 | 代码质量、安全 | 安全/性能/生产就绪 |
| 典型触发词 | "审查项目""架构有没有问题""整体一致性" | "code review" | "生产就绪" |
| 适用阶段 | 设计/重构/集成/重大变更 | 每次 PR | 上线前/定期审计 |

> **三技能不互斥，是不同粒度的审查。** 一次大型重构可能同时用三个。

---

## 三步方法论总览

```
Step 1: 横向自洽性检查 ── 组件间是否一致
    ↓ (找出矛盾点)
Step 2: 纵向架构深度审查 ── 职责是否各司其职
    ↓ (评估架构质量)
Step 3: 持续改进审查 ── 哪些可增强/精简/优化
    ↓ (输出改进路线图)
```

每步都是**独立可用**的，但串联使用最系统。

---

## Step 1: 横向自洽性检查

**核心问题**：系统的各个组件之间是否存在矛盾——接口契约对不上、字段命名不一致、假设互相冲突。

### 检查清单

| 类别 | 检查项 | 参考 |
|------|--------|------|
| **接口契约** | 模块A声称的输入/输出，与模块B实际调用的参数类型/命名是否一致？ | (ATAM: 场景模块 / arc42: §3 Context) |
| **字段对齐** | 同一概念在不同层（如 API/DTO/DB）的字段名/类型/可选性是否一致？ | (arc42: §5 Building Block) |
| **命名一致** | 核心实体（如 `User`）的命名在所有引用点是否统一？ | (arc42: §12 Glossary) |
| **配置假设** | 各模块对环境/配置/权限的假设是否互相矛盾？ | (ATAM: 质量属性 / arc42: §2 Constraints) |
| **错误处理约定** | 各模块的错误码/异常类型/重试策略是否一致？ | (arc42: §6 Runtime View) |
| **状态机一致** | 跨模块的同一对象状态定义是否相同？ | (ATAM: 场景) |
| **数据流向** | 数据从输入到持久化的路径上，各环节对数据结构的假设是否一致？ | (arc42: §5 Building Block) |
| **协议版本** | 客户端/服务端的 API 版本、序列化格式是否对齐？ | (ATAM: 性能/可用性) |

### 检查步骤

1. 列出系统的核心组件（如 API 层、领域层、基础设施层、外部服务）
2. 对每个**跨组件边界**，检查上表所有检查项
3. 记录每个矛盾点：**预期 vs 实际**，影响范围
4. 按影响范围排序：阻塞型（运行时会崩）> 隐藏型（边缘场景）> 优雅降级型

### 输出格式

```
## 自洽性检查报告
- [CRITICAL] API 层的 User.id 类型是 string，DB 层是 int → 类型不一致
- [HIDDEN] 订单状态的"已取消"在模块A是 CANCELLED，在模块B是 CANCELED
- [MINOR] 日志格式约定在用户服务是 JSON，在订单服务是 key=value
```

## Step 2: 纵向架构深度审查

**核心问题**：系统的分层、职责、链路是否合理？每个组件是否做且只做它该做的事？

### 检查清单

| 类别 | 检查项 | 参考 |
|------|--------|------|
| **职责单一** | 每个模块/类/函数是否只做一件事？是否存在"上帝类"或"上帝模块"？ | (ATAM: 可修改性 / arc42: §5 Building Block) |
| **分层清晰** | 依赖方向是否单向（如 UI→Service→Domain→Infra）？是否有反向依赖？ | (ATAM: 性能/可维护性 / arc42: §4 Solution Strategy) |
| **链路完整** | 一个核心用例（如"下单"）的端到端链路是否能画出来？有没有断点？ | (ATAM: 场景 / arc42: §6 Runtime View) |
| **抽象恰当** | 抽象层级是否一致？有没有"上帝接口"或"过度抽象"？ | (ATAM: 可修改性) |
| **质量属性** | 性能/可用性/安全/可修改性在架构中是否被显式处理？ | (ATAM 核心) |
| **决策可追溯** | 重要架构决策是否记录了理由（ADR）？ | (arc42: §9 Architectural Decisions) |
| **风险可见** | 已知技术债和风险是否被记录和追踪？ | (arc42: §11 Risks) |
| **部署合理** | 架构是否与部署模型一致（微服务/单体/无服务器）？ | (arc42: §7 Deployment View) |

### 检查步骤

1. 画出系统的**组件图**（按层次而非按功能）
2. 对每个组件检查"职责单一"和"抽象恰当"
3. 对每条**跨层调用**检查依赖方向
4. 选 3-5 个**核心用例**，逐个追踪端到端链路
5. 对每条链路检查质量属性是否被处理
6. 检查 ADR 和技术债记录

### 输出格式

```
## 架构深度审查报告
- 职责: ⚠️ OrderService 同时处理订单逻辑、支付调用、消息发送 → 拆分
- 分层: ✅ 依赖方向单向，无反向
- 链路: ❌ "取消订单"用例在 OrderService 后断点，库存回滚缺失
- 质量: ⚠️ 可修改性低，OrderService 修改常触发 PaymentService 改动
- 决策: ❌ 5 个重要决策无 ADR
```

## Step 3: 持续改进审查

**核心问题**：当前系统中哪些可以增强、精简或优化？

### 检查清单

| 类别 | 检查项 | 参考 |
|------|--------|------|
| **冗余代码** | 是否存在功能重复的代码/类/模块？是否能合并？ | (arc42: §5 Building Block) |
| **过时文档** | 文档/SKILL.md/注释是否与实际代码一致？ | (ATAM: 可维护性) |
| **废弃功能** | 是否有已标记废弃但仍在使用/导出的功能？ | (ATAM: 可修改性) |
| **可强化点** | 哪里缺少错误处理？缺少测试？缺少日志？缺少验证？ | (ATAM: 可用性/安全) |
| **简化机会** | 是否有过度设计、过度抽象、可以简化的部分？ | (arc42: §4 Solution Strategy) |
| **性能瓶颈** | 是否有明显的 N+1、重复计算、不必要的 IO？ | (ATAM: 性能) |
| **安全加固** | 是否所有输入都验证？敏感操作都审计？ | (ATAM: 安全) |
| **可观测性** | 关键路径是否有日志/指标/追踪？ | (ATAM: 可用性 / arc42: §6) |

### 检查步骤

1. 从 Step 1/Step 2 的报告出发，收集**改进候选**
2. 按"投入产出比"排序：高价值低投入 > 低价值高投入
3. 区分**立即做**（安全/阻塞）、**下迭代做**（性能/增强）、**技术债登记**（非紧急但需追踪）
4. 写入项目级 TODO 或 issue 跟踪

### 输出格式

```
## 改进路线图

### 🔴 立即做（安全/阻塞）
- [ ] 修复 OrderService 职责拆分（影响所有订单相关功能）
- [ ] 补全"取消订单"库存回滚（当前存在数据不一致风险）

### 🟡 下迭代做（增强）
- [ ] 为核心服务添加结构化日志
- [ ] 缓存热点数据（订单详情、商品信息）
- [ ] 补充 5 个核心场景的集成测试

### 🟢 技术债登记
- [ ] 5 个架构决策缺少 ADR
- [ ] 用户服务的密码加密逻辑需要迁移到统一 crypto 模块
```

## 执行模式

### 快速模式（15-30 分钟）

适用于：日常小变更后、项目交付前

```
Step 1: 自洽性检查 → 只检查接口契约 + 字段对齐（约 10 个检查点）
Step 3: 改进审查 → 只记录立即要做的（不超过 5 条）
```

### 完整模式（1-3 小时）

适用于：重大重构前、新模块集成、季度审查

```
完整执行三步，每步的完整检查清单都过一遍
```

### 深度模式（1-2 天）

适用于：年度架构审查、并购评估

```
三步 + ATAM 质量属性场景 + arc42 12 节全面文档
由多个 ZCode session 并行执行不同组件的审查
```

## 使用场景

| 场景 | 推荐模式 | 关注重点 |
|------|---------|---------|
| 新模块接入项目 | 快速 | Step 1（接口契约） |
| 重大重构前 | 完整 | 全部三步 |
| 定期项目健康检查 | 完整 | Step 2 + Step 3 |
| 性能问题排查 | 快速 | Step 2 的质量属性 + Step 3 的性能 |
| 安全审计 | 完整 | Step 1 + Step 2 + ATAM 安全参考 |
| 团队入职项目了解 | 完整 | 用 review 过程倒推系统全貌 |

## 使用方式

### 在 ZCode 中

```
加载 system-review 技能，然后：
"请用 system-review 的快速模式审查当前项目，重点是刚接入的支付模块"
"请用 system-review 的完整模式做一次季度审查"
```

### 与 code-reviewer 配合

- `code-reviewer` → 单 PR/单文件的代码层审查
- `system-review` → 项目/模块/系统的整体层审查
- 两者互补：code-reviewer 发现代码异味时，可用 system-review 看是否反映系统层面的设计问题

### 与其他技能配合

| 配合技能 | 配合方式 |
|---------|---------|
| `brainstorming` | 用 system-review 发现问题后，用 brainstorming 探索解决方案 |
| `writing-plans` | 改进路线图确定后，用 writing-plans 写实施计划 |
| `code-reviewer` | system-review 发现代码层问题后，转 code-reviewer 深入 |
| `subagent-driven-development` | 深度模式可拆分为多个 ZCode session 并行审查不同组件 |

---

## 局限

- **不替代代码级审查** — 代码异味、bug、安全漏洞走 code-reviewer 或 production-code-audit
- **不替代实际测试** — 改进路线图中的"补全测试"需要 testing-patterns 配合
- **不替代人工判断** — system-review 的输出是 ZCode 的分析，最终决策仍需用户确认
- **不适用于紧急 hotfix** — 紧急修复场景走 systematic-debugging

## 相关技能

- `code-reviewer` — 单文件/单 PR 的代码质量审查
- `production-code-audit` — 全量生产级质量审计
- `agent-skill-architecture` — 技能本身的设计规范
- `brainstorming` — 发现问题后探索方案
- `writing-plans` — 确定改进方向后写实施计划
- `subagent-driven-development` — 多组件并行审查
