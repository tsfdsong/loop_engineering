# [F]/[H]/[P] 标注规范

> 本文档定义 evidence-first 技能的三类标注规则。
> 每条论述中的**每一句话**都必须标注。

---

## 标注总览

| 标注 | 名称 | 含义 | 优先级 | 验证要求 |
|------|------|------|:------:|---------|
| **[F]** | Fact（事实） | 已通过证据验证 | 🟢 最高 | 必须可追溯 |
| **[H]** | Hypothesis（假设） | 基于经验但未验证 | 🟡 中 | 必须明确标注 |
| **[P]** | Principle（原则） | 通用工程参考 | 🔴 最低 | 不作为论据 |

**冲突优先级：F > H > P**

---

## [F] 事实（Fact）

### 定义
**已通过 git log / 文件内容 / 文档 / 命令验证**的陈述。

### 要求

1. **可追溯**：必须能指出"在哪个文件的哪一行"或"哪个命令的输出"
2. **可复现**：其他人在相同环境跑相同命令能得到相同结果
3. **不夸大**：事实是事实，不要"为了论证"修饰

### 标注格式

```
[F] v5.4 是 v6.0 的回滚基线
  └─ 证据: ls skills/skill-hub/SKILL.md.v5.4.backup（文件存在）
  
[F] v54-baseline.json 含 27 条黄金轨迹
  └─ 证据: cat tests/golden-traces/v54-baseline.json | python -c "import json,sys; print(len(json.load(sys.stdin)))"
  
[F] 当前 main 分支最新提交是 cedb61c
  └─ 证据: git log -1 --oneline
```

### 禁止用法

| 错误 | 正确 |
|------|------|
| [F] v5.4 是老版本（有大量用户） | [H] v5.4 可能有用户（未验证）|
| [F] 软件工程通常保护向后兼容 | [P] 软件工程通常保护向后兼容（通用原则）|
| [F] 这个项目很成功 | [F] 项目有 X 个 star，Y 个 fork（具体数据）|

---

## [H] 假设（Hypothesis）

### 定义
**基于通用经验或合理推测，但未经过本项目验证**的陈述。

### 要求

1. **明确标注**：不能伪装成 [F]
2. **说明依据**：基于什么经验/什么类比
3. **可证伪**：必须能通过具体证据推翻
4. **谨慎用词**：用"可能""或许""通常""一般"等

### 标注格式

```
[H] v5.4 可能有用户依赖
  └─ 依据: 基于通用经验，老版本通常有人用
  └─ 验证方式: 查 telemetry / 用户调研
  └─ 可证伪: 如果有数据显示 v5.4 用户 < 1%，假设推翻

[H] 删除 baseline 可能影响回归测试
  └─ 依据: baseline 通常用于回归保护
  └─ 验证方式: 跑 test_v54_baseline_27_traces_unchanged
```

### 何时升级为 [F]？

- 跑过验证命令并得到结果 → 升级为 [F]
- 查过 git/docs 找到证据 → 升级为 [F]
- 用户明确告知事实 → 升级为 [F]

---

## [P] 原则（Principle）

### 定义
**通用工程原则、最佳实践、行业惯例**，与本项目**无关**。

### 要求

1. **明确标注为通用参考**：不要套用到本项目而伪装成事实
2. **说明来源**：来自哪个原则/哪个流派
3. **谨慎使用**：仅作参考，不作为论据

### 标注格式

```
[P] SemVer 规定次版本号应向后兼容
  └─ 来源: semver.org 规范
  └─ 注意: 这是通用原则，本项目是否遵守需独立验证

[P] 软件工程通常保留向后兼容以保护用户
  └─ 来源: 通用最佳实践
  └─ 注意: 不适用于所有项目，特别是基线版本
```

### 何时升级？

- 如果原则在本项目被**明确违反** → 标注为 [P] 并说明"本项目例外"
- 如果原则被本项目**明确遵循** → 可以基于本项目事实升级为 [F]
- 否则保持 [P]，不作为本项目结论的依据

---

## 冲突处理（最重要）

### F vs H

```
[F] v5.4 SKILL.md 备份今天创建（git log 验证）
[H] v5.4 可能是老版本（基于通用经验）

冲突解决: [F] 胜出 → 结论: v5.4 是今天定稿的版本
```

### F vs P

```
[F] v5.4 不是老版本（git log 验证）
[P] 软件工程通常保护向后兼容

冲突解决: [F] 胜出 → 结论: 通用原则在本项目不适用
```

### H vs P

```
[H] v5.4 可能有用户（未验证）
[P] 软件工程通常保护向后兼容

冲突解决: 都是推测，但 [P] 范围更广，[H] 更具体
         → 选择 [H] 作为本项目的假设（更贴近具体）
         → 标注: "虽然通用原则支持兼容，但本项目 v5.4 实际角色需查证"
```

## 标注覆盖率要求

| 场景 | 标注覆盖率 |
|------|----------|
| **事实清单段** | 100%（必须全部 [F]） |
| **分析论述段** | ≥ 90% |
| **结论段** | ≥ 95% |

**未标注的句子 = 不可信**。

---

## 反模式（禁止）

| 反模式 | 后果 |
|--------|------|
| 把 [H] 写成 [F] | 错误事实化 |
| 把 [P] 写成 [F] | 套模板 |
| 不标注任何东西 | 整段不可信 |
| 用"应该""可能"无标注 | 套用经验掩盖事实 |
| 把多个事实混在一句 | 难以追溯 |

---

## 自检问题

完成论述后自问：

- [ ] 每句话都标注了吗？
- [ ] [F] 有追溯证据吗？
- [ ] [H] 明确标注了吗？未伪装成 [F]？
- [ ] [P] 明确是通用原则吗？未套用到本项目？
- [ ] F/H/P 之间有冲突吗？按优先级处理了吗？
- [ ] 结论可追溯到 [F] 吗？

**全部 ✅ = 论述可信**。
