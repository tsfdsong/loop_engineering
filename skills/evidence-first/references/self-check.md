# 自检 4 问（详解）

> 本文档是 evidence-first 技能"自检 4 问"的详细说明。
> 任何长篇论述（> 5 段）**必须**先答这 4 问。

---

## 自检 4 问

### 问 1：我有 [F] 事实依据吗？

#### 意图
强制你"先查证再论述"。

#### 判断标准
- ✅ 至少 1 个 [F] 标注，且有可追溯的证据
- ⚠️ 全是 [H] 或 [P] → 论述**不可信**
- ❌ 无任何标注 → 禁止论述

#### 答"No"时怎么办
```
1. 回到事实清单
2. 跑 5 项必查命令
3. 找到至少 1 个 [F]
4. 再回来论述
```

#### 答"Yes"时
继续问 2。

---

### 问 2：[H] 假设明确标注了吗？

#### 意图
防止"假设伪装成事实"。

#### 判断标准
- ✅ 每个 [H] 都有明确标注
- ✅ 用"可能""或许""通常"等谨慎用词
- ❌ 用了"显然""毫无疑问""一定是" → 隐藏假设

#### 常见伪装

| 伪装 | 实际 |
|------|------|
| "这个项目肯定有用户" | [H] 可能有用户，未验证 |
| "显然应该保留" | [P] 通用原则认为应该保留 |
| "一般而言 X" | [P] 通用经验 |
| "大家都知道" | [H] 部分人可能知道，未在本项目验证 |

#### 答"No"时怎么办
```
1. 找出所有"显然""肯定""一般"等词
2. 改为谨慎用词 + 标注 [H]
3. 重新过一遍论述
```

---

### 问 3：错了损失大吗？

#### 意图
高损失场景必须有 [F] 主导，不能靠 [H]/[P] 推断。

#### 损失分级

| 损失等级 | 场景 | 证据要求 |
|---------|------|---------|
| 🔴 极高 | 删除/破坏性操作、合并冲突、回滚 | **必须 [F] 主导**，否则不行动 |
| 🟡 中 | 设计决策、选型建议、重构 | **至少 50% [F]**，可辅以 [H]/[P] |
| 🟢 低 | 解释说明、对比分析、文档编写 | 标注清楚即可 |

#### 示例

🔴 **高损失**（如"要不要删 v5.4 baseline"）：
```
需要:
- [F] 当前有 X 个用户使用 v5.4（telemetry 数据）
- [F] 删除会破坏 X 个测试用例（grep 验证）
- [F] 替代方案已就绪（验证替代方案存在）

不需要:
- [H] "可能"有人用
- [P] "通常"应该保护向后兼容
```

🟢 **低损失**（如"为什么 v5.4 兼容有价值"）：
```
可以:
- [F] v54-baseline.json 有 27 条 trace
- [H] 这 27 条用于回归保护
- [P] 回归测试是好实践

但需要明确标注，不能基于"用户生态"等无证据的 H/P 论述。
```

#### 答"高损失但证据不足"时怎么办
```
1. 主动停止论述
2. 告知用户: "这个决策损失大，但我证据不足，建议先查 X"
3. 不要凭推断行动
```

### 问 4：能说"我不清楚"吗？

#### 意图
**最重要的问题**。不确定性是诚实，不是弱点。

#### 判断标准
- ✅ 真的不确定 → **说"我不清楚"**
- ✅ 不确定 + 用户问 → 主动问 / 查
- ❌ 不确定 + 强行回答 → 错误

#### "我不清楚"的正确用法

| 场景 | 错误 | 正确 |
|------|------|------|
| 用户问"v5.4 有多少用户" | 套用"用户生态"论述 | "我不清楚，需要 telemetry 数据" |
| 用户问"为什么保留 baseline" | 套用"通常应该保留" | "让我先查 baseline 的实际用途" |
| 不确定通用原则 | 套用模板 | "通用原则是 X，但本项目是否适用需查证" |
| 不确定细节 | 编造细节 | "细节我不清楚，主要事实是 Y" |

#### 答"No"（不能说"我不清楚"）时
- 这通常意味着**过度自信**或**面子问题**
- 重新评估：诚实 > 显得专业

---

## 4 问执行流程

```
开始论述前
  ↓
答 4 问（即使心里答，不写出也可）
  ↓
全部 YES → 开始论述
任一 NO  → 处理:
  ├── 问 1 NO → 补查事实
  ├── 问 2 NO → 标注 [H]
  ├── 问 3 NO + 证据不足 → 停止，告知用户
  └── 问 4 NO → 重新评估，可能需要承认不知道
```

## 自检表（可视化）

```
□ 问 1: 至少 1 个 [F] 标注
□ 问 2: [H] 明确标注，未伪装
□ 问 3: 损失大时有 [F] 主导
□ 问 4: 能说"我不清楚"

全部 ✅ → 可以论述
任一 ❌ → 先处理再论述
```

## 失败案例（2026-06-29 v5.4 事故）

### 当时自检会怎样？

```
□ 问 1: 至少 1 个 [F] 标注
   ❌ 没有 —— 全是"用户生态""SLA"等无证据论述

□ 问 2: [H] 明确标注
   ❌ 没有 —— 把通用原则当事实

□ 问 3: 错了损失大吗
   🟡 中 —— 删除决策，需要 [F] 主导
   但当时全是 [H]/[P]

□ 问 4: 能说"我不清楚"
   ❌ 没有 —— 编造"用户""SLA"等来避免承认不知道

4 问全 ❌ → 仍然长篇论述 → 事故
```

### 改进后

```
□ 问 1: 至少 1 个 [F] 标注
   ✅ [F] v5.4 SKILL.md 备份今天定稿（git log 验证）

□ 问 2: [H] 明确标注
   ✅ [H] v5.4 可能还有用户（基于通用经验，未验证）

□ 问 3: 错了损失大吗
   🟢 低 —— 只是解释性问题，不是删除决策

□ 问 4: 能说"我不清楚"
   ✅ "v5.4 的实际角色需查证，不能基于通用原则强答"

4 问中 3 个 ✅ + 1 个改进 → 短答案 + 标注 [F]/[H]/[P] → 正确
```

---

## 总结

> **自检 4 问 = 长篇论述的"刹车系统"**
>
> 4 问中任一失败 = 减速/停止/重新查证
> 4 问全通过 = 安全进入论述
>
> **没有经过 4 问的长篇论述 = 高风险操作**
