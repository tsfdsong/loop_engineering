# 仅补验收（可选 · 轻量 A+D）

> **定位（薄执行环）**：不再是默认模式的「核心 intake」。  
> **仅当**裸 `/loop` 缺可测验收时，补一短清单后进入编码；**默认审计闸门，不 Ask「需求理解是否正确」**。  
> `--auto` / go 子任务：**禁止**用本引擎补做需求分析——缺验收则失败回交。

## 何时启用

| 场景 | 行为 |
|------|------|
| 裸 `/loop`，验收齐全 | 跳过本文件 |
| 裸 `/loop`，验收缺失/过糊 | 补短清单 → 继续编码（默认不 Ask） |
| `/loop --auto` 或缺 goal | **不启用**；失败回交（见 mode-auto.md） |
| 明显未定型（选型 / 要不要做） | 提示 brainstorming /go，不补验收开干 |

## A: 补验收短清单

### 输入
- 用户功能描述
- 项目上下文（技术栈、已有模块、架构约束）

### 输出
```
验收条件清单:
1. [G4] 积分累积 API 正确响应（单元测试+集成冒烟）
2. [G4+G6] 积分字段使用 decimal(10,2）（前后端一致）
3. [G4+G5] 积分明细查询响应 < 200ms
4. [F1+F4] 前端页面控制台 0 错误，交互流程正确
```

每条尽量绑定门禁维度。无法绑定 = 标记「模糊」——若整体仍无法执行，提示上游 brainstorming/go，而非开长确认流。

### BDD 示例（可选，默认不 Ask）

需要自检理解时，可对关键条件生成 1 个 BDD 示例供内部对照；**不**强制弹出确认：

```
验收条件 1: 积分累积 API 正确

场景: 正常累积
  Given 用户 A 当前有 100 积分
  When  触发累积事件 (+50.5 积分)
  Then  余额 = 150.5
        AND 积分明细表新增 1 条记录
```

## D: 自信度（降级）

纯化后默认走 **审计闸门**：

```
🟢 默认 → 审计闸门（自动采纳补齐结果，继续编码）
  - 用户已明确的条件
  - 模板可套用的通用/API/前端等短清单

例外：补完后仍无法形成可执行目标 → 停止并提示 brainstorming /go
（不再用 🟡/🔴 Ask 做「需求理解确认」主路径）
```

执行态仍需 Ask 的场景（断点、合并、自愈 stagnated、验证官 BLOCKED 等）见 mode-default / mode-auto，与「补验收」无关。

## 验收条件→门禁维度映射

| 验收条件类型 | 默认门禁维度 |
|-------------|-------------|
| 功能正确性 | G4 + G5 |
| 性能要求 | G4(性能测试) + G5(真实压测) |
| 前端交互 | F1 + F3 + F4 |
| 数据类型一致 | G4 + G6 |
| 构建/部署 | G7 + G8 |
| 环境连通 | G1 |
