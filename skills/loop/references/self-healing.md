# 自愈闭环

门禁矩阵出现 ❌ 时统一处置引擎。按根因三分、分级触发。

## 根因分类

| 类型 | 判定特征 | 处置分支 | 调 systematic-debugging |
|------|---------|---------|:---:|
| **A. 被测代码 bug** | 堆栈指向业务代码 / 渲染不符预期 / 交互逻辑错 | 修业务代码 → 重跑门禁 | ✅ |
| **B. 验证脚本本身错** | ref 失效 / 选择器定位失败 / 操作步骤遗漏 | 修验证步骤 → 重验 | ❌ |
| **C. 环境问题** | 服务没起 / 端口不通 / 登录态过期 | 修环境(试1轮) → 修不好才汇报 | ❌ |
| **🎨 主观项 (F5)** | 配色/间距/风格等设计差异 | 标记"设计待确认"，列入报告，不自动改 | ❌ |

**关键**: 错误分类错误是最大的 token 浪费源——把 B/C 当成 A 修，会反复改业务代码却修不好。

## 分级触发处置

```
门禁矩阵有 ❌
    │
    ▼
对每条失败项 → 根因归类:
    │
    ├── 🔴 G1(依赖不可达)
    │     → 三级回退: docker-compose → daemon → 阻塞汇报
    │     → 不调 debugging（这是环境问题不是代码bug）
    │
    ├── 🔴 硬错误(G2-G8)
    │     → 全自动 systematic-debugging → 修代码 → 重跑
    │
    ├── 🟡 A类(业务bug)
    │     → 自动 debug+修 → 重跑
    │
    ├── 🟡 B类(验证脚本错)
    │     → 修验证步骤 → 重验（不调debugging，代码没错）
    │
    ├── 🟡 C类(环境)
    │     → 修环境(试1轮) → 修不好才汇报
    │     → 优先 docker-compose 起本地依赖
    │
    ├── 🟡 F5客观项(区块缺失/元素类型错)
    │     → 自动 debug+修 → 重验
    │
    └── 🟢 F5主观项(配色/间距/风格)
          → 标记"🎨 设计待确认" → 列入报告 → 不自动改
```

## 终态定义

| 终态 | 触发 | 行为 |
|------|------|------|
| **success** | 门禁全绿 + 验收条件全部通过 | 进入 Step⑥ 交付验证 |
| **stagnated** | 连续 2 轮同一问题无进展 | AskUserQuestion → 用户决策 |
| **exhausted** | 达到用户确认轮次 ×1.5 仍未通过 | AskUserQuestion → 用户决策 |

### stagnated 确认

```
问题: "连续2轮 [问题描述] 无进展，如何处置？"

┌─────────────────────────────────────────────────┐
│ 【推荐】换实现路径重试                             │
│   问题可能在技术方案本身，换不同库或模式再试1轮     │
│ 【备选】降复杂度分步交付                           │
│   先交付可独立验证的子集，剩余部分拆分到独立任务    │
│ 【备选】继续1轮不变                               │
│   再试一次当前策略（可能临时环境问题）              │
│ 【备选】终止本次任务                               │
│   保留已完成的代码在隔离分支，不合并                │
└─────────────────────────────────────────────────┘
```

### exhausted 确认

```
问题: "已达推荐轮次上限，仍有 [N] 条验收条件未通过，如何处置？"

┌─────────────────────────────────────────────────┐
│ 【推荐】追加 [X] 轮                                │
│   剩余条件已有部分进展（[证据]），追加预计可达标     │
│ 【备选】调整验收条件                               │
│   重新审视未通过的条件是否合理/必要                 │
│ 【备选】终止保留分支                               │
│   [N]个文件已提交到[分支名]，可手动完善后合并        │
│ 【备选】切换实现策略                               │
│   当前方案可能走不通，换完全不同的技术路线           │
└─────────────────────────────────────────────────┘
```

## 不降级红线

- 禁止静默删减/跳过验收条件中的任何功能点
- 禁止擅自降低验收标准
- 禁止遇到难点绕开/"以后再做"
- 禁止在门禁矩阵未全绿时宣称"任务完成"
- 环境不通≠验证通过（G1红线）
- console/network ❌ 不许放过
- 截图仅留证：结构化数据才是判据

## 经验沉淀

自愈修复成功后 → 即时沉淀（详见 `references/experience-library.md`）：
- 检查经验库是否已有 → 无则提取教训
- 写入对应领域文件 → 报告标注"📍 新增经验"
