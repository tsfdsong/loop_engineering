# loop 默认模式（🟢🟡🔴 自适应）

loop 默认模式——用户输入 `/loop 功能，验收条件` 时触发。根据复杂度评估自动选择轻量/标准/完整模式。

---

## 验收条件推理（A+D 引擎）

当用户没有提供验收条件或条件过于模糊时，使用 A+D 引擎自动推理（详见 `references/acceptance-inference.md`）：

1. **自动推理**：根据功能类型匹配默认验收条件模板
2. **BDD 示例生成**：输出具体示例确认理解
3. **自信度闸门**：
   - 🟢 高自信 → 审计闸门，直接继续
   - 🟡 中自信 → AskUserQuestion，5s倒计时
   - 🔴 低自信 → BDD 示例验证，用户快速选

### 默认验收条件模板

| 功能类型 | 关键词 | 自动补全的默认验收条件 |
|----------|--------|----------------------|
| **通用** | 所有功能 | ① 单元测试通过（覆盖率 ≥ 80%）② 无回归 Bug |
| **API/接口** | API、接口、端点、REST | + 响应时间 < 200ms、+ 错误码规范、+ 输入参数校验 |
| **数据库** | 数据库、表、迁移、存储 | + 数据完整性校验、+ 迁移可回滚、+ 无 N+1 查询 |
| **前端/UI** | 页面、组件、UI、前端 | + 组件可访问性、+ 响应式适配、+ 加载/空/错误三态 |
| **性能优化** | 性能、加速、优化、慢 | + Benchmark对比、+ 无内存泄漏、+ P99 < 目标值 |
| **安全相关** | 安全、权限、加密、认证 | + OWASP检查、+ 敏感数据不落盘、+ 权限边界测试 |
| **重构** | 重构、重写、改进、清理 | + 行为不变(回归通过)、+ 圈复杂度下降 |

## 执行流程

```
/loop 触发
    │
    ▼
Step ⓪ 断点续跑检测
  • 检查 .loop-state-<分支slug>.json
  • 存在 → AskUserQuestion:
    问题: "检测到未完成的loop任务：[功能名]，如何处理？"
    ┌─────────────────────────────────────────────────┐
    │ 【推荐】从断点恢复                               │
    │   已完成需求确认和计划拆分，恢复后可继续执行      │
    │ 【备选】放弃并重新开始                           │
    │   将重新确认需求、验收条件和拆分计划              │
    └─────────────────────────────────────────────────┘
  • 不存在 → 新建状态文件，进入 Step ①
  │
  ▼
Step ① 解析 + 验收条件 + 需求理解
  • 提取功能描述、验收条件、技术栈
  • 验收条件缺失 → 触发 A+D 引擎自动推理 BDD 示例
  • 输出【需求理解总结 + 验收条件清单 + BDD 示例】
  • AskUserQuestion:
    问题: "需求理解是否正确？验收条件是否齐全？"
    ┌─────────────────────────────────────────────────┐
    │ 【推荐】确认开工                                 │
    │   理解正确，验收条件覆盖了核心场景                 │
    │ 【备选】修改某条条件                             │
    │   想调整某条验收条件的具体值                      │
    │ 【备选】补充遗漏场景                             │
    │   需要补充未覆盖的业务场景或约束条件              │
    └─────────────────────────────────────────────────┘
  🔒 未得到确认 → 绝不进入下一步
  │
  ▼
Step ② 复杂度评估 + 轮次确认
  • 6维度评分（0/1/2 分，加权求和）:
    需求清晰度×2 / 影响范围×2 / 预估代码量×1 / 
    风险等级×2 / 依赖复杂度×1 / 用户角色数×1
  • ≤6→🟢轻量 / 7-14→🟡标准 / ≥15→🔴完整
  • 轮次推荐: 🟢1~2 / 🟡2~4 / 🔴3~6
  • AskUserQuestion:
    问题: "复杂度评估为 [模式]，采用多少轮？"
    ┌─────────────────────────────────────────────────┐
    │ 【推荐】[N] 轮                                   │
    │   基于6维度评分([X]分)：[理由]                    │
    │ 【备选】[N-1] 轮                                 │
    │   缩减1轮，合并部分验证步骤，适合需求很明确的场景  │
    │ 【备选】[N+1] 轮                                 │
    │   增加1轮深度验证，适合全新架构或高风险的场景      │
    └─────────────────────────────────────────────────┘
  │
  ▼
Step ③ 计划拆分（按模式裁剪）
  🟢 轻量：1-3条简明任务清单
  🟡 标准：writing-plans 完整拆分
  🔴 完整：writing-plans 全量 + 依赖分析 + 工作量评估
  │
  ▼
Step ④ Git 隔离（遵循 using-git-worktrees）
  • 检测当前是否已在隔离环境
  • 已在 worktree → 复用
  • 在普通 checkout → git worktree add loop/<slug>-<MMDD>
  │
  ▼
Step ⑤ 闭环编码（详见 references/gate-matrix.md + references/self-healing.md）
  每 Round = Implement → GitCheck → 门禁矩阵 → 自愈闭环 → Decide
  
  • 门禁矩阵: G1-G8 + F1-F5（按任务特征启用）
  • 自愈闭环: 门禁失败 → A/B/C/🎨 分级触发
  • Decide:
    ✅ 门禁全绿 → success，进入交付
    🟡 连续2轮无进展 → stagnated → AskUserQuestion
    🔴 exhausted → AskUserQuestion
  │
  ▼
Step ⑥ 交付验证（按模式裁剪）
  🟢 轻量：verification 快速验证
  🟡 标准：code-reviewer + verification
  🔴 完整：code-reviewer + verification + to-prd
  
  经验复盘沉淀 → loop-library/lessons/
  
  AskUserQuestion:
    问题: "所有验收条件已通过，如何处理代码？"
    ┌─────────────────────────────────────────────────┐
    │ 【推荐】合并到主分支                             │
    │   门禁全绿(G4:XX%, G5:全2xx, F1:0错误)，回归通过 │
    │ 【备选】保留分支                                 │
    │   代码保留在 [分支名]，供手动审查后合并            │
    │ 【备选】丢弃改动                                 │
    │   放弃所有修改，回到原始分支状态                  │
    └─────────────────────────────────────────────────┘
  │
  ▼
✅ 完成
```

## 模式对比

| 维度 | 🟢 轻量 (≤6分) | 🟡 标准 (7-14) | 🔴 完整 (≥15) |
|------|:--:|:--:|:--:|
| 断点续跑 | ✅ | ✅ | ✅ |
| A+D 验收推理 | ✅ | ✅ | ✅ |
| 需求确认 | AskUserQuestion | AskUserQuestion | AskUserQuestion |
| 计划拆分 | 1-3条清单 | writing-plans | 全量+依赖分析+垂直拆分 |
| Git 隔离 | ✅ | ✅ | ✅ |
| 推荐轮数 | 1~2 | 2~4 | 3~6(无硬上限) |
| 门禁矩阵 | ✅ G1-G8 | ✅ G1-G8+F1-F4 | ✅ G1-G8+F1-F5 |
| 自愈闭环 | ✅ 分级触发 | ✅ 分级触发 | ✅ 分级触发 |
| 前端验证 | 涉及前端时强制 | 涉及前端时强制 | 涉及前端时强制 |
| 代码审查 | ⏭️ | ✅ code-reviewer | ✅ code-reviewer |
| 文档输出 | ⏭️ | ⏭️ | ✅ to-prd |
| 不降级红线 | ✅ | ✅ | ✅ |
| 合并决策 | AskUserQuestion | AskUserQuestion | AskUserQuestion |

## 手动覆盖

```
/loop --lite 修复分页Bug     → 强制 🟢 轻量
/loop --full 重构用户模块      → 强制 🔴 完整
/loop --standard 添加导出功能  → 强制 🟡 标准
```
