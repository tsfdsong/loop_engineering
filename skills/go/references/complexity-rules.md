# L0 复杂度评估规则(纯规则,零 token)

> 编排层第一步:判断任务走 L1/L2/L3 哪一档。**零 token 成本**(不调 LLM,纯关键词+信号匹配)。
> 设计原则:不为"省 token"而花 token;误差靠硬闸门+门禁验收纠正。

---

## 评估信号

| 信号 | 如何获取 | L1 直通 | L2 标准 | L3 完整 |
|------|---------|:---:|:---:|:---:|
| **影响文件数** | 用户描述/grep 预估 | =1 | 2-5 | >5 |
| **任务关键词** | 从功能描述提取 | typo/改字符串/加注释 | API/测试/重构/单功能 | 跨模块/架构/新系统 |
| **是否跨工具** | 是否同时涉及前后端+审查 | 否 | 否 | 是 |
| **用户显式指定** | 命令标志 | `/go-fast` | (默认) | `/go-full` |

---

## 判定逻辑(按优先级,首个命中生效)

```
Step 1: 显式指定优先
  用户带 -fast 标志  → L1 (跳过后续规则)
  用户带 -full 标志  → L3 (跳过后续规则)

Step 2: 关键词强匹配 L1
  功能描述命中 L1 词表 AND 影响文件数=1 → L1

Step 3: 跨工具判定
  同时涉及 前端+后端+审查 → L3

Step 4: 文件数判定
  影响文件数 >5 → L3
  影响文件数 2-5 → L2

Step 5: 关键词判定
  命中 L3 词表 → L3
  命中 L2 词表 → L2

Step 6: 默认兜底
  无法明确判断 → L2 (让硬闸门纠正)
```

---

## L1 关键词词表(直通档)

```
typo, 错别字, 改字符串, 加注释, 改文案, 修拼写,
改常量, 改配置值, 改提示语, 改变量名, 修缩进, 改格式
```

**特征**: 单文件、无逻辑变更、无新依赖、纯文本/配置修改。

---

## L2 关键词词表(标准档)

```
新增API, 写测试, 单元测试, 重构函数, 优化查询, 加字段,
单功能, 接口, 端点, 服务, 中间件, 钩子, 工具函数, 修Bug(单点)
```

**特征**: 单模块内、有逻辑变更但范围可控、可能1-2个新依赖。

---

## L3 关键词词表(完整档)

```
跨模块, 架构, 重构(大), 新系统, 重写, 迁移, 改造,
集成, 联调, 新功能(复杂), 平台, 框架, 全栈, 端到端
```

**特征**: 多模块、影响范围广、需要多工具协作、风险等级高。

---

## 模糊地带处理

约 20-30% 任务无法明确分级(如"优化登录功能"):
- **默认走 L2 标准档**(不上不下,安全)
- 在硬闸门确认时展示评估理由,用户可选 [调整级别]
- L3 误判为 L1 → 门禁验收会失败(L1基础验证太弱挡不住复杂任务),触发升级
- L1 误判为 L3 → 用户在硬闸门看到"小题大做",选 [调整] 降级

**结论**: 规则的误差可以被硬闸门和门禁验收纠正,不需要 L0 本身做到 100% 准确。

---

## 与 loop 复杂度评估的区别

| 维度 | loop Step② | 编排层 L0 |
|------|-----------|----------|
| 方法 | 6维度评分(0-2分加权求和) | 纯规则关键词+信号匹配 |
| Token成本 | 需花 token 推理 | **零 token** |
| 粒度 | 细(6维度) | 粗(4信号) |
| 适用 | 单工具闭环(需精确) | 多工具调度(够用即可) |

**为什么编排层用更粗的规则**: 编排层的核心价值是"调度多工具",不是"精确评估"。L0 只需快速分流,精确度交给硬闸门和门禁。
