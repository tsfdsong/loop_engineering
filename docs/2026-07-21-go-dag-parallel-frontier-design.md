# Design: go DAG 并行前沿 + 机械脚本层（无角色注册表）

> **日期**: 2026-07-21  
> **状态**: **Approved** · 2026-07-21  
> **动机**: 墙钟优先加速「写码 + 测修」；借鉴宿主 Subagent 的真并行与隔离，**不**引入专职角色注册表。  
> **取代/收敛**: 否决「全 skill Subagent 化」与「精选角色注册表」作为加速主路径。

## 0. 核心目的（验收北极星）

在 LoopEngine 插件内，让用户通过 **`/go`（及既有 loop 闭环）** 更快完成：

1. **写码（B）**：独立模块/子任务 **真并行** 实施  
2. **测修（C）**：独立失败域 **真并行** 排查与修复  

约束：

- **墙钟优先**（总 token 允许略升）  
- **高效**：能脚本完成的不派 agent；任务包保持短小（V3）  
- **优雅**：一个调度概念（DAG 并行前沿），Subagent 只是执行器  
- **明确不做**：Subagent **角色注册表**（loop/CR/system-review 等「品牌化角色」产品）

## 1. 问题陈述

- 宿主（Cursor / Claude / ZCode）已提供 Subagent/Task 真并行；本插件已有 `/go` family/DAG、worktree、dispatch 协议，但 **「何时一次派齐可并行节点」** 与 **「测修失败如何按域并行」** 仍偏文档约定、默认不够狠。  
- 把方法论 skill 做成「角色注册表」主要提升组合叙事，**不直接**提升墙钟；串行专职链甚至变慢。  
- 机械门禁（测、审计、smart_commit）若仍靠模型「记得跑」，浪费墙钟与 token。

## 2. 已确认决策

| # | 决策 | 选择 |
|---|------|------|
| D1 | 速度 vs token | **墙钟优先** |
| D2 | 加速重点 | **写码并行 + 测修并行**（非需求调研主路径） |
| D3 | 主架构 | **DAG 并行前沿 + 机械脚本层** |
| D4 | Subagent | **仅执行器**；由 go 按图调度，不设角色注册表 |
| D5 | 角色注册表 | **取消 / 不做** |
| D6 | 审查类 skill | 保持现有 skill 加载或按门禁偶尔派 Task；**不**产品化为固定角色 ID |

## 3. 架构

```text
                    ┌─────────────────────────┐
                    │  /go 编排（调度中心）      │
                    │  family → DAG → 前沿计算  │
                    └───────────┬─────────────┘
                                │
              ┌─────────────────┼─────────────────┐
              ▼                 ▼                 ▼
        可并行节点集合      汇合点/依赖边       机械门禁
        （写码 / 测修）      （等齐再继续）      （脚本，非 agent）
              │                                     │
              ▼                                     ▼
     宿主 Task/Subagent                      smart_commit /
     + worktree（V4）                        测试命令 / audit…
              │
              ▼
         短任务包（V3 五类输入）
         内嵌需要的 SKILL 指针（非角色表）
```

**一句话**：go 算「现在能并行跑谁」→ 一次派齐执行器 → 机械步骤走脚本 → 汇合后再算下一前沿。

## 4. DAG 并行前沿（写码 B）

### 4.1 行为

1. Step 0 family + 现有 `dag-rules` / 拆任务得到节点与依赖。  
2. **并行前沿** = 入度为 0（或依赖已完成）且 `side_effect` 互不冲突（无共享写集 / 已 worktree 隔离）的节点集合。  
3. 对前沿内每个 **写码节点**：创建独立 worktree + 一份短任务包 → 宿主并行派执行器（现有 Cursor/其它 dispatch 协议）。  
4. 执行器运行 **loop 闭环逻辑**（通过任务包引用 `skills/loop`，**不是**「loop 角色注册项」）。  
5. 全部完成或失败策略触发后 → 合并/交接 → 计算下一前沿。

### 4.2 冲突与降级

- 写集重叠或无法 worktree 隔离 → **禁止**同前沿，改串行。  
- 宿主无法多 Task → 降级顺序执行（已有 degradation 叙事，保持）。  
- L1 / 单节点 → 不并行，直通 loop（避免调度税）。

### 4.3 验收

- ≥2 个无依赖写码子任务时，go **默认尝试**并行派发（非仅文档建议）。  
- 状态/进度能看出「前沿并行中」（V5）。

## 5. 测修并行（C）

### 5.1 触发

- 测试门禁或多文件失败，且失败可分成 **独立域**（不同包/用例/子系统，无共享失败根因假设）。  

### 5.2 行为

1. 按域切片（文件集 / 用例前缀 / 模块）。  
2. 每域一个短任务包（systematic-debugging 或 loop-fix 指针 + 复现命令）。  
3. **并行**派执行器；禁止多域改同一 worktree。  
4. 汇合后跑 **一次** 聚合验证（脚本跑测试）。  

### 5.3 不触发并行

- 失败明显同源、或修复会互相覆盖 → 单执行器串行。  

## 6. 机械脚本层（高效）

原则：**agent 不负责「记得做机械步骤」**。

| 步骤 | 首版挂钩 |
|------|----------|
| 智能提交（防漏 untracked） | 已有 `scripts/smart_commit.py` / `/git-commit` |
| 部署/插件健康 | 已有 `audit_tools.py` / `/audit` |
| 测试闸门 | go/loop 汇合点 **必须**跑项目测试命令（脚本/CI 本地入口），结果写状态文件 |
| 可选后续 | 格式化/lint 同样脚本化 |

审查（code-reviewer / system-review）**不**改成注册角色；L2/L3 交付前按现有 G9/G10 叙事：需要时派 Task 或同会话加载 skill——由门禁级别决定，不新增注册表。

## 7. 任务包（执行器输入）

保持 V3 五类：scope / goal / constraints / format / context。  

**禁止**：把完整主会话历史塞进子执行器。  
**允许**：指向具体 `SKILL.md` 路径或「先 Read 某 skill」的一句指令——这是 **文档引用**，不是角色 ID。

## 8. 非目标

- Subagent 角色注册表、角色 marketplace、统一 `agent_type=code-reviewer` 产品层  
- 需求调研阶段默认多路 Subagent 狂派（非本次 B/C 重点）  
- 保证总 token 下降  
- 重写全家 skill 为 agent 进程  

## 9. 组件与改动面（实施时）

| 区域 | 改动 |
|------|------|
| `skills/go/SKILL.md` + `dag-assembly` / 调度相关 | 写清「并行前沿」算法与默认开启条件 |
| `dag-rules` / family 或拆任务输出 | 如需：节点标注 `parallel_safe` / `write_set`（最小字段） |
| dispatch 协议（如 cursor-dispatch） | 确保前沿多节点可一次多派 |
| loop 失败/测试路径 | 多域失败 → 并行测修切片 |
| 文档 `using-loopengine` | 一句话：加速靠 DAG 并行 + 脚本，无角色表 |
| **不改** | 新建 `agents/registry.yaml` 之类 |

## 10. 测试与成功标准

1. 构造 2 个无依赖写码任务：日志/状态显示并行派发（或测试 mock dispatch）。  
2. 两域测试失败夹具：并行两个 fix 包，汇合后全绿。  
3. 写集冲突夹具：拒绝同前沿并行。  
4. L1 单任务：不引入并行调度税。  
5. 仓库内 **不存在** 角色注册表文件/文档承诺。  

墙钟：在双独立任务场景下，相对严格串行有可观测缩短（环境受限时至少「调度意图与结构」可测）。

## 11. 反选项

| 方案 | 否决理由 |
|------|----------|
| 精选/全量角色注册表 | 用户明确不要；非速度主杠杆 |
| 仅文档鼓励并行、调度不改 | 达不到「实施」 |
| `/fast` 新命令 | 与 go 重叠，YAGNI |
| loop 内默认同目录多写手并行 | 违反 V4，不优雅 |

## 12. 实施分期

| 期 | 内容 |
|----|------|
| **P0** | go 写码前沿并行默认化 + 冲突检测 + 文档 |
| **P1** | 测修多域并行 + 汇合脚本验证 |
| **P2** | 机械门禁清单补齐（测试入口约定、与 smart_commit 衔接） |

详细任务拆解：本 spec **Approved** 后由 spec-driven-development 或按用户指示直接 `/loop` 实施。
