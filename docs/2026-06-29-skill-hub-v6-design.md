**版本**：v1.0（brainstorming 阶段定稿）
**日期**：2026-06-29
**作者**：brainstorming 技能（ZCode MiniMax-M3）
**状态**：待 spec self-review + 用户审阅

## 一、背景与动机

### 1.1 问题陈述

skill-hub 当前 v5.4 实现"**每次只加载一个技能**"的单技能路由模型（见 `skills/skill-hub/SKILL.md` 核心规则 #2、关键指令 #2、自身描述）。

**现实冲突**：
- 真实工程任务常包含**多个意图**（调研+讨论+审查+分析+决策）
- v5.4 的"冲突裁决"机制**只处理"竞争"关系**（多技能选 1），不处理"互补"关系
- 典型的"MCP 代码探索 vs 原生工具"分析任务触发了 5 类意图关键词，skill-hub 只路由到 `system-review`，错过了 `brainstorming` 和 WebSearch 调研

### 1.2 用户痛点（2026-06-29 实证）

用户原话：
> "对于一个复杂的涉及多个技能的任务时候，skill-hub 的技能路由策略太过单一，无法处理复杂任务，涉及多技能使用的场景。"

### 1.3 行业调研结论

| 来源 | 关键启示 |
|------|---------|
| Anthropic《Building Effective Agents》 | 明确推荐 **Orchestrator-Workers 模式**（中心 LLM 动态拆解、委派、综合）；但**警告**保持简单、按需升级 |
| Lilian Weng Agent 综述 | MRKL/HuggingGPT/API-Bank 都是多专家协同；**LLM 自我评估有盲点**（ChemCrow 案例：LLM 自评与人类评估结论相反） |
| 本项目 `loop` 技能 | 已是"复合技能"（内含 9 个 references + 1 个脚本），其内部"reference 加载"模式可借鉴 |

**核心结论**：引入 Orchestrator 模式是**行业共识**，但**必须保持简单**且**防御 LLM 评估盲点**。

## 二、设计目标与非目标

### 2.1 目标

1. **后向兼容 100%**：现有 53 个技能的 SKILL.md 零修改
2. **复合任务自动识别**：5 类典型复合任务模式开箱即用
3. **Orchestrator 编排能力**：串行/并行编排、上下文隔离、循环防御
4. **可观测性**：每次编排有 trace_id，可随时回溯
5. **可回滚**：任何阶段可一键回退到 v5.4 行为

### 2.2 非目标

1. **不替代 loop 技能**：loop 是复合技能的实例，skill-hub v6.0 是其底层支撑
2. **不修改 53 个现有技能的 SKILL.md**
3. **不引入新的元技能类型**（如 agent vs skill 的区分）
4. **不实现 LLM 自我评估的"完全准确"**——只能防御其盲点
5. **不直接调度 MCP 工具**（如 WebSearch）：MCP 工具由 LLM 在 skill 执行过程中自行调用，skill-hub 只调度 skill；Orchestrator 不感知 MCP 工具的存在

## 三、详细设计

### 3.1 三层架构

```
┌──────────────────────────────────────────────────────┐
│ Layer 1: 意图分析层 (Intent Analysis)                 │
│   ├─ 解析用户输入                                    │
│   ├─ 识别意图类型（1-N 个）                          │
│   └─ 输出: 意图向量 [调研, 讨论, 审查, 决策, ...]    │
├──────────────────────────────────────────────────────┤
│ Layer 2: 复杂度评估层 (Complexity Evaluator)         │
│   ├─ 单技能任务（意图数=1）→ 走 v5.4 单技能路径       │
│   ├─ 复合任务（意图数≥2）→ 走 Orchestrator 路径       │
│   └─ 输出: 任务类型 + 候选技能集                      │
├──────────────────────────────────────────────────────┤
│ Layer 3: 编排层 (Orchestrator)                       │
│   ├─ 仅在复合任务时激活                               │
│   ├─ 任务拆分 → 技能调度 → 结果综合                   │
│   └─ 防御: LLM 自评盲点 + 循环检测 + 强制停止         │
└──────────────────────────────────────────────────────┘
```

### 3.2 5 类复合任务类型

| # | 类型 | 典型场景 | 默认技能链 | 编排方式 | 触发关键词 |
|---|------|---------|----------|---------|----------|
| 1 | 调研+决策 | "对比 A 和 B 方案的优缺点，给出选择" | `brainstorming` → `system-review` → `writing-plans` | 串行 | 调研 + 决策/选型/对比 |
| 2 | 分析+建议 | "审查这个项目，给出改进意见" | `system-review` → `brainstorming` | 串行 | 审查/分析 + 改进/建议 |
| 3 | 诊断+修复 | "这个 Bug 是什么引起的，修复它" | `systematic-debugging` → `verification-before-completion` | 串行 | 报错/Bug + 修复 |
| 4 | 设计+实现 | "设计并实现 XX 功能" | `brainstorming` → `writing-plans` → `executing-plans` | 串行 | 设计 + 实现/开发 |
| 5 | 规划+并行 | "调研 A、B、C 三个库并对比" | `subagent-driven-development` | 并行 | 并行/多任务 + 调研 |

### 3.3 复杂度评估（混合策略）

**第一层：规则判定**（零 token 成本）
- 关键词扫描：复用现有 53 技能表
- 规则模式匹配：5 类复合任务的触发条件
- 单技能（意图数=1）→ 直接走 v5.4 路径
- 复合任务（意图数≥2）→ 启动 Orchestrator

**第二层：LLM 验证**（仅在规则冲突或模糊时）
- 触发条件：规则冲突 / 规则无匹配但 LLM 判断为复合
- 防御 LLM 自评盲点：列出 Top-2 类型用 `AskUserQuestion` 让用户选
- 不允许 LLM 单独决定

### 3.4 Orchestrator 编排协议

#### 3.4.1 串行 vs 并行

| 复合任务类型 | 编排方式 | 依据 |
|------------|---------|------|
| 调研+决策 | 串行 | 后一步依赖前一步输出 |
| 分析+建议 | 串行 | 建议必须基于分析结果 |
| 诊断+修复 | 串行 | 修复必须先定位 Bug |
| 设计+实现 | 串行 | 实现依赖设计 |
| 规划+并行 | **并行** | 子任务相互独立 |

#### 3.4.2 上下文隔离策略

**默认 Mode B（累积传递）**：
- 每个 skill 加载时只看到前序 skill 的**最终结论摘要**（≤ 500 token）
- **不看**前序 skill 的中间步骤（避免 context 污染 + 节省 token）
- 摘要必须包含：结论 + 关键证据 + 不确定项

**模式对照**（Mode A/C 作为 v7.0 扩展候选）：
| 模式 | 行为 | 适用 |
|------|------|------|
| Mode A 完全隔离 | 各 skill 独立结论 | v6.0 默认的"独立任务"备选 |
| **Mode B 累积传递** | **前序结论摘要传递** | **v6.0 默认** |
| Mode C 选择性共享 | Orchestrator 决定哪些事实传递 | v7.0 候选 |

#### 3.4.3 强制停止条件

```yaml
强制停止条件:
  最大步数: 5 步（防复合任务无限展开）
  时间上限: 10 分钟（防 hang 住）
  重复检测: 连续 2 步意图向量相同 → 强制中止
  验证闸门: 每个 skill 输出必须通过 verification-before-completion

中止后行为:
  - 抛出结构化错误给用户
  - 列出已完成步骤 + 未完成步骤
  - 询问用户: 重试 / 跳过未完成步 / 降级到单技能模式
```

#### 3.4.4 失败处理矩阵

| 失败类型 | 处理 |
|---------|------|
| 规则路由失败 | 降级到 LLM 兜底（同 v5.4） |
| LLM 验证失败 | 列出 Top-2 候选，AskUserQuestion 让用户选 |
| Skill 加载失败 | 跳过该步，记录错误，继续下一步 |
| Skill 执行失败 | 中止编排，问用户决策 |
| Token 预算超限 | 触发 headroom 压缩前序摘要 |
| 循环调用 | 强制中止，提示用户 |

### 3.5 可观测性

每次编排后自动输出 trace（**实现时建议 JSON 格式，YAML 仅作示例**）：

```yaml
trace_id: <UUID>
detected_intents: [调研, 决策]
task_type: 调研+决策
skills_invoked: [brainstorming, system-review, writing-plans]
total_tokens: ~15000
per_skill_tokens: {brainstorming: 8000, system-review: 5000, ...}
per_skill_duration: {brainstorming: 3min, system-review: 2min, ...}
stop_reason: completed | timeout | loop_detected | user_abort
```

**用户可随时输入**：`/trace <trace_id>` 看到完整编排过程。

## 五、风险登记

| 风险 | 概率 | 影响 | 缓解措施 |
|------|:---:|:---:|---------|
| 复合识别误判（简单任务被升级） | 中 | 中 | Phase 1 灰度 + 用户反馈循环 |
| Token 开销爆炸 | 低 | 高 | 强制停止 + 摘要压缩 + 用户可中止 |
| LLM 自评盲点导致错误编排 | 中 | 高 | 规则判定优先 + LLM 仅在规则冲突时启用 |
| 用户困惑 | 高 | 中 | trace 可见性 + 一句话告知每步 |
| 现有 53 技能被破坏 | 低 | 高 | 回归测试 + 灰度期默认关闭 |
| MCP 并发调用冲突 | 低 | 中 | Orchestrator 内部串行化 MCP 调用 |
