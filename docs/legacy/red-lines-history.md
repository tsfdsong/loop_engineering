### 9.1 方案选型（4 条）— 论源：brainstorming / clean-code / software-architecture

- **R1.1 反选项清单** [🔴] — 任何方案推荐前必须先列 ≥ 2 个"被否决备选"+ 否决理由；禁止"只推荐 A 不说为什么不选 B"
- **R1.2 机会成本可见** [🟠] — 推荐方案必须说明"放弃了什么换取什么"（trade-off 透明）
- **R1.3 YAGNI** [🔴] — 禁止为"未来可能用到"提前抽象 / 预留接口 / 加配置开关
- **R1.4 可回滚红线** [🟠] — 禁止 all-in 不可逆选择（DB schema 无回滚 / 协议硬编码无版本）

### 9.2 技术选型（1 条）— 论源：software-architecture

- **R2.4 PoC / Spike 时间盒** [🔴] — 重大选型必须 spike（≤ 4h），超时强制决策；禁止"边想边写大段代码"

### 9.3 修复原则（3 条）— 论源：systematic-debugging / code-reviewer

- **R3.1 根因分析** [🔴] — 禁止症状级修复（吞异常 / 改判断绕过 / 加默认值掩盖）；必须先找根因再修
- **R3.3 修复 scope = bug scope** [🟠] — 禁止"顺手重构"+ unrelated cleanup 进 fix commit（PR 纯净原则）
- **R3.5 同根 Bug 扫描红线** [🔴] — 修复 bug 前必须扫描同根变体：1) 根因定位到机制层；2) 全仓搜索同机制所有调用点；3) 列出全部受影响位置；4) 一次性修复全部 OR 登记 deferred + 引用 issue 跟踪

### 9.4 重构原则（2 条）— 论源：refactoring

- **R4.1 测试保护** [🔴] — 无测试覆盖代码禁止重构（先补测试，再重构）
- **R4.4 机械重构优先** [🟠] — rename / extract function / move class 优先于架构重构（先小后大）

### 9.5 小步快跑 / 简洁高效（6 条 · 核心）⭐ — 论源：clean-code / brainstorming / code-reviewer / evidence-first

- **R5.1 KISS 红线** [🔴] — 禁止为"优雅 / 通用 / 扩展性"增加不必要的抽象层 / 装饰器 / 元编程
- **R5.2 减法红线** [🔴] — 新增 ≥ 5 行代码前必须先回答"能不能不写？能不能复用现有？"
- **R5.3 Tracer Bullet 红线** [🔴] — 新功能必须先打通端到端骨架（walking skeleton），再补血肉；禁止"完整设计 + 完整实现"一气呵成

- **R5.6 Working > Comprehensive** [🟠] — 禁止"完整文档先行"；先让代码 work，再补文档
- **R5.7 Spike & Iterate** [🟠] — 不确定时先写 spike（探索代码）→ 再写实现；禁止"边想边写大段代码"

### 1.10 🆕 测试纪律（v1.3.2 加入 · 教训来自 L#002 specs 卡死事故）

> **背景**：2026-07-03 specs 安装卡死事故。AI 助手在 5 轮本地测试中**每次都加 `--skip-specs` 跳过**真正的问题，导致"端到端验证通过"但用户实际命令卡死 1 powershell + 7 个 git 子进程，**永不退出**。详细事故链见 `docs/lessons-learned.md` L#002。
>
> **核心混淆**：`git pull` 在远端不存在时**不是 fail** 而是 **hang**（反复重试 TCP/认证）。`if ! cmd` 抓不到，`2>/dev/null` 吞不掉。必须 `timeout N` 主动设上限。

**5 条硬规则**（任何 release/fix PR 涉及 install 或 deploy 改动前必查）：

1. **黄金路径测试纪律**：每个 release/PR 必跑**用户最小命令**（零 flag）端到端至少 1 次。开发期 `--force` / `--skip-specs` / `--only` 等是便利**不是替代**。
2. **区分 fail vs hang**：
   - **失败（sync fail）**：返回非 0 → `if ! cmd` 能捕获
   - **挂起（hang）**：永不返回 → 必须 `timeout N cmd` 主动设上限
   - `2>/dev/null` / `2>&1 | Out-Null` **只吞错误信息，不终止挂起**
3. **测试时禁用开发期 flag**：若加 flag 才"测试通过"，立刻怀疑"用户裸跑会怎样"。"测试通过 ≠ 真实可用"。
4. **脏状态 = 测试失败**：本地有遗留 `~/.xxx/` 脏目录（指向不存在 URL）时跑 deploy 100% 触发 hang。**测试前清环境**应成为肌肉记忆。
5. **加新功能时 red team 自己的设计**：
   - 依赖仓库不存在？远端返回 5xx？网络慢到 timeout？用户在防火墙后？
   - **任何一个没设计降级路径的依赖都是定时炸弹**
   - 例：5 轮前 specs 外部化功能，**5 轮没人问"如果 specs 仓库不存在会怎样"**

**自查清单**（每次写 install/deploy 脚本前）：

- [ ] 是否引入了新的外部依赖（git clone、API、远程服务）？
- [ ] 外部依赖不可达时，脚本是 fail（可控）还是 hang（用户卡死）？
- [ ] 是否给了"如果依赖不存在怎么办"的降级路径（timeout / 默认跳过 / 完全删除）？
- [ ] 本地测试是否至少跑过 1 次"用户裸命令"（无任何 flag）？
- [ ] 本地测试前是否清掉脏目录（`rm -rf` 模拟首次安装）？

### 6.2 §3.6 自查清单（5 项 · 会话结束前必查 · v6.10）

> 原位于 AGENTS.md L243-251。

- [ ] **本轮决策点是否全部用 `AskUserQuestion` 触发？**（v6.10 新增 · 关键项）
- [ ] **每个 AskUserQuestion 调用是否含 2-4 个选项 + 1 个标注 `(推荐)`？**
- [ ] **不推荐选项的描述是否说明了不推荐理由？**
- [ ] **是否避免了"你觉得呢？""还需要什么？"等开放式追问？**
- [ ] **是否避免了"建议 X"但又不调 AskUserQuestion 的混合形式？**（最常见的违规模式）

> **违规判定**：决策点用文字列出 + 不调 AskUserQuestion = 🔴 红线违规，必须自愈。

### 6.3 §3.7.1 §3.6 自查清单第 6 项（v6.11 新增）

> 原位于 AGENTS.md L262-264。

- [ ] **本轮"建议 / 如需 / 后续步骤 / 候选方案"段落是否含 ≥ 2 个候选？如有，是否全部用 AskUserQuestion 触发？**（v6.11 新增 · 防止"建议 + 列表"逃逸）

#### 3.7.2 违规判定

- 建议段落含 ≥ 2 个候选但未调 AskUserQuestion = 🔴 红线违规（与"决策点"违规同级）
- 单一推荐（如 `**建议 X，因为 Y**`）= 合规（无选项列表，不触发 AskUserQuestion）
- 纯事实陈述 / 引用 / 清单（无选项性内容）= 合规

#### 3.7.3 6 条避免方法（v6.11 升级）

1. 写 "## 🎯 决策点" 之前先自问 → **立即调** AskUserQuestion（不要先写 markdown 再调工具）
2. 红线 3.6 五项 checklist 每次输出结束前必走
3. "如需..."型句子默认触发 AskUserQuestion
4. 区分"建议"与"决策"的写法：
   - **建议 + 单一推荐**（合规）：`**建议 X，因为 Y**`
   - **建议 + 多候选**（违规 → 改用 AskUserQuestion）：禁止 markdown 列表
   - **决策 + 多选项**（必须 AskUserQuestion）
5. 养成"先调 AskUserQuestion 再写 markdown 选项"的肌肉记忆
6. **🆕 自检触发条件**：任何输出末尾含"如 / 或 / 可以 / 推荐 / 建议 / 候选" + 选项列表 → 必走 AskUserQuestion
