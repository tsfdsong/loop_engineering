# LoopEngine skill-hub v6.1 设计文档（2026-06-29）

> **本设计文档**：v6.0 → v6.1 协同重构的设计方案。
> **核心目标**：在 v6.0 复合任务编排基础上，显式建模 go / loop / subagent-dd 三技能协同契约，
> 抽取 shared 共享基础设施，桥接 subagent-dd 审查能力为 go/loop G9/G10 增强。

---

## 1. 背景与动机

### 1.1 现状

v6.0 已建立复合任务编排（Orchestrator 模式），但 go / loop / subagent-dd 三技能的关系**未在调度表中显式建模**：

- go（编排层）调用 loop（执行层）通过 `--auto` 模式
- subagent-dd（平行范式）独立于 Orchestrator 5 类复合任务执行
- go/loop 与 subagent-dd 的审查能力**没有桥接**
- go/loop 之间的 owner 字段、原子写算法**概念重复**

### 1.2 问题

1. **生态位不清晰**：用户不清楚何时选 go、何时选 loop、何时选 subagent-dd
2. **能力不互通**：subagent-dd 的 spec/code 双审能力无法被 go/loop 复用
3. **代码重复**：owner 字段定义、原子写算法在 go/loop 两侧重复

### 1.3 目标

1. **不合并三技能**——保留 go / loop / subagent-dd 各自生态位
2. **显式建模协同契约**——skill-hub 调度表 + skill-relationships.md
3. **抽取 shared/ 共享基础设施**——5 份 spec + 1 个 Python 函数
4. **桥接 subagent-dd 审查能力**——6 个核心契约 + opt-in 灰度

## 3. 关键设计决策

### 3.1 不合并三技能（保持独立）

**理由**：
- **抽象层不同**：go 编排层 / loop 执行层 / subagent-dd 平行范式
- **范式不同**：loop = 自动化门禁 / subagent-dd = 人工子代理审查
- **合并破坏分层**：单文件 50KB+，范式冲突，用户选择困惑

**结论**：三技能各占一个不可替代的生态位，**保留独立**。

### 3.2 维持 state-protocol 双轨制

**理由**：
- go 宏观（任务树）vs loop 微观（单任务）是有意为之的分层
- 合并会破坏分层设计
- 既有 .orchestrate-state.json / .loop-state-*.json 100% 兼容

**结论**：双轨制**保留**，但共享 owner 字段规范、原子写规范。

### 3.3 shared/ 抽取策略

**抽取范围**：高+中优先级（5 份 spec + 1 个 Python 函数）

| 优先级 | 抽取内容 | 文件 |
|--------|---------|------|
| 高 | owner 字段定义 | `shared/references/owner-field-spec.md` |
| 高 | 原子写规范 | `shared/references/atomic-write-spec.md` + `scripts/atomic_write.py` |
| 中 | 状态机通用规范 | `shared/references/state-protocol-base.md` |
| 中 | 断点恢复基础协议 | `shared/references/breakpoint-recovery-base.md` |
| 中 | G9/G10 协作契约 | `shared/references/g9-g10-coordination.md` |

**不抽取的内容**（避免过度设计）：
- `state-protocol.md` 两份整文（双轨制保留）
- `complexity-rules.md` vs `mode-default.md`（方法论不同）
- `gate-matrix.md` 整文（loop 独占）
- `degradation.md` vs `self-healing.md`（职责正交）

### 3.4 subagent-dd 桥接深度

**选择方案 A 最小桥接**（6 个核心契约）：

```python
# 6 个桥接函数
dispatch_implementer           # 派遣实现者（4 状态枚举）
dispatch_spec_reviewer         # 派遣规格审查（✅/❌）
dispatch_code_quality_reviewer # 派遣质量审查（3 层 Issues）
model_select                   # 模型选型信号
handle_implementer_status      # 4 状态应对动作
review_gate                    # 强顺序约束
```

**理由**：
- 6 个核心契约已覆盖 G9/G10 增强需求
- 不实现方案 B 的元能力（避免过度设计）
- 不实现方案 C 的 SOP DSL（实现复杂度过高）

### 3.5 桥接 opt-in 灰度

```bash
# 默认（不启用）
export LOOPENGINE_BRIDGES=disabled    # 默认值

# 启用桥接（alpha）
export LOOPENGINE_BRIDGES=alpha       # 加载 bridges/contract.py
```

**铁律**：
- 默认关闭（100% 兼容 v5.4/v6.0）
- 启用时**不改变** go/loop 原有 G9/G10 默认实现
- 桥接失败时**自动降级**到原 G9/G10，不报错中断

## 8. 总结

v6.1 重构本质是**「在 v6.0 已建立的复合任务编排基础上做治理」**：
- 不发明新概念，只把已有概念**显式化、共享化、桥接化**
- 100% 向后兼容，通过 opt-in 灰度
- 工作量适中（17h），风险可控
- 落地后：3 技能生态位更清晰，跨技能复用度提升，subagent-dd 的审查能力可被 go/loop 选用
