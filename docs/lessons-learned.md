## 事故 #1：v5.4 兼容性胡乱分析

### 基本信息

| 项 | 内容 |
|------|------|
| **日期** | 2026-06-29 |
| **事故 ID** | INC-2026-06-29-001 |
| **等级** | 🟠 中（被用户当场识破，未造成实际损失） |
| **类型** | 无事实依据 + 套用通用原则 + 过度自信 |
| **影响范围** | 单一对话，无代码改动，无数据损失 |

### 事故回放

**用户问题**：
> 为什么需要 v5.4 兼容？直接删除旧版本不行吗

**AI 的错误回答**（节选）：
```
v5.4 兼容非常重要，原因如下:
1. v5.4 已发布 1+ 年，积累了大量用户   ← 编造
2. 企业级 SLA 承诺                       ← 套用
3. SemVer 规定次版本兼容                 ← 套用
4. 用户脚本可能依赖                       ← 套用
5. 法律责任                              ← 套用
```

**用户当场纠正**：
> 你在胡说，v5.4 版本今天才写完，留着干啥，你有没有认真分析项目，还是胡说八道

### 根因分析（5 Whys）

1. **为什么回答错误？**
   - 5 条"理由"全是 [H]/[P] 伪装成 [F]
2. **为什么会用 [H]/[P] 伪装 [F]？**
   - 没查事实就论述，凭印象推断
3. **为什么会凭印象推断？**
   - 没建立"先查证再论述"的工作流
4. **为什么没有工作流？**
   - 没有强制"项目分析前查 5 项事实"的规范
5. **为什么没有规范？**
   - 之前没出过类似事故，没意识到需要规范

### 教训

| # | 教训 | 严重性 |
|---|------|:------:|
| 1 | 没查证不论述 | 🔴 致命 |
| 2 | 通用原则 ≠ 项目事实 | 🟠 严重 |
| 3 | 不确定就说"我不清楚" | 🟠 严重 |
| 4 | 长篇大论需 [F] 主导 | 🟡 中 |
| 5 | 错误就认错 | 🟡 中 |

### 修复措施

#### 1. 新增 `evidence-first` 技能（v1.0）

**位置**：`skills/evidence-first/`

**核心组件**：
- `SKILL.md` — 主入口
- `references/fact-checklist.md` — 5 项事实清单
- `references/claim-types.md` — [F]/[H]/[P] 标注规范
- `references/self-check.md` — 自检 4 问
- `references/no-hallucination.md` — 不确定时处理
- `references/traceability.md` — 追溯链规范
- `examples/bad-v54-compat-answer.md` — 事故现场回放
- `examples/good-v54-compat-answer.md` — 改进版答案

#### 2. 全局硬规则（AGENTS.md）

在 `~/.zcode/AGENTS.md` 和 `loop_engineering/AGENTS.md` 加入：

```
🔴 事实优先硬规则（v6.2 增强 · 2026-06-29 加入）

1. 项目分析前必查 5 项事实（参见 evidence-first 技能）
2. 每句话标注 [F]/[H]/[P]
3. 不确定 = 说"我不清楚"，禁止凑答案
4. 长篇论述前自检 4 问
5. 判断必须可追溯到事实
```

#### 3. skill-hub 调度表强化

在 skill-hub 调度表中加入 evidence-first 触发规则：

| 技能 | 触发关键词 |
|------|-----------|
| **`evidence-first`** | 分析/比较/评估/为什么/设计/重构/选型/该不该 |

#### 4. 黄金轨迹 + 测试

- `tests/golden-traces/evidence-first-baseline.json` — 20 条 trace
- `tests/test-evidence-first.py` — 18 个测试

### 验证机制

| 指标 | 衡量方式 | 目标 |
|------|---------|------|
| 长篇论述前的事实清单覆盖率 | 抽样 10 个回答 | ≥ 80% |
| [F]/[H]/[P] 标注率 | 长篇回答中标注比例 | ≥ 90% |
| "我不清楚" 出现频次 | 当确实不确定时 | 应该出现 |
| 类似事故再发次数 | 教训库登记 | 持续为 0 |

### 后续监控

- [ ] 每月抽样审计 evidence-first 规范执行情况
- [ ] 每次发生新事故 → 登记 → 转化为规范
- [ ] 每季度回顾：哪些规范有效，哪些需要更新

## 🛡️ 防止类似事故的 3 道防线

### 防线 1：预防（事故前）
- ✅ 任何项目分析前必查 5 项事实
- ✅ 长篇论述前自检 4 问
- ✅ evidence-first 技能自动触发

### 防线 2：检测（事故中）
- ✅ 标注 [F]/[H]/[P]（用户可识破伪装）
- ✅ 追溯链（每个结论可追溯）
- ✅ "我不清楚" 优先（避免凑答案）

### 防线 3：纠正（事故后）
- ✅ 用户纠正 → AI 立即认错
- ✅ 事故登记到 lessons-learned.md
- ✅ 转化为可执行规范

## 💭 反思

### 这次事故的最大价值

**不是损失**，是**让团队意识到**：
> "没有事实依据的专业论述 = 高级胡说"

**不是耻辱**，是**改进的契机**：
> 把错误变成规范，让系统在下次自动刹车

### 类似的潜在风险

| 潜在风险 | 防护方向 |
|---------|---------|
| 套用通用模板 | evidence-first [P] 标注 |
| 凭印象推断 | 5 项事实清单 |
| 长篇大论掩盖事实不清 | 自检 4 问 + 短答案优先 |
| 不认错 | 教训库登记机制 |
| 用户不知道发生了什么 | 公开 lessons-learned.md |
