## 2. 回答（追加到本文件）

### A1：实际跑的可用性
**答**：实际**不会失败**，但行为与文档描述的"自动串行"不一样。真实情况是：
1. `/composite 1` 触发 → 走 LLM 验证路径
2. LLM 列出 Top-2 候选（如 "brainstorming" 和 "system-review"）
3. **你必须用 AskUserQuestion 选 1 个**
4. 选完后只跑那 1 个，**不会自动跑后续链**

所以"自动串行"是**未来目标**，现在**是手动串行**。

### A2：LLM 验证的具体算法
**答**：文档没说 LLM 验证的具体算法，但根据 `composite-task-types.md` 的"硬约束"反推：
- **输入**：用户输入 + 53 技能关键词表的"命中的关键词"
- **输出**：Top-2 候选技能 + 理由
- **核心约束**：必须列出 2 个（不允许 1 个或 3 个）— 防御 LLM 自评盲点
- **失败处理**：如果 LLM 自己也选不出来，提示用户手动选

**这是一个未实现的接口**——alpha mock 阶段的 LLM 验证是"规则模拟 + LLM fallback"，**不**是真正的 LLM 推理。

### A3：决策树的具体触发
**答**：按决策树第 1 步判断：
- "有现成 writing-plans 计划吗？" → **否**（用户临时说的）
- "跨模块/需要 DAG/需要 worktree 真并发？" → **是**（分页 + 权限通常在不同模块）
- → 走 **`go`**

具体流程：
1. `/go 实现分页 + 权限`
2. go Step ①.5 需求分析（6 维度）
3. Step ②-④ 拆任务、写 plan
4. Step ⑤ 调 `loop --auto` 在 worktree 内闭环
5. Step ⑦ 全局回归
6. Step ⑦.5 system-review G10

### A4：性能预算的实际意义
**答**：**目前没意义**——alpha mock 阶段的所有"性能预算"都是**设计目标**，不是实测。原因：
- LLM 验证频繁触发时，token 开销会显著高于 5%
- alpha mock 本身用"规则模拟 + LLM fallback"，规则模拟零成本但 fallback 高成本
- 真正能验证要等 Plan Orchestrator 真实实现后

**对用户的建议**：把性能预算当作"未来期望"而非"当前能力"。

### A5：v5.4 黄金轨迹的具体内容
**答**：**我也不知道**——这是 10-search.md 引用 SKILL.md 原文时提到的，但 SKILL.md 没列出 27 条的具体内容。**这是本调研的一个空白**（已记录在"局限 3"）。

如果需要这部分信息，应该：
1. 直接读 `tests/golden-traces/v54-baseline.json`（v5.4 的黄金轨迹文件）
2. 列出 27 条的具体场景描述
3. 评估每条是否被 v6.0/v6.1 破坏
