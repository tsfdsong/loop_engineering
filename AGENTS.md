## 🔴 1. MCP 红线规则（输入端 · 本项目最高优先级 · 不可违反 · v6.4 重构 · 精简版）

> **任何需要理解代码结构的操作，必须先用 MCP 工具，禁止直接 Read 全文件。**
> 此规则适用于本项目（loop_engineering）的所有开发、调研、分析工作。

### 1.1 适用范围

- 修改代码、调研代码、解释代码、分析架构
- 查找函数/类/变量定义、查找引用位置
- 了解项目结构、浏览目录、理解文件内容
- **只要目的是"理解代码"，就必须 MCP 优先**

### 1.2 探查策略（6 场景分层 · v6.11 重构 · 2026-07-01）

**核心原则**：按工作场景分层定义 MCP 使用规则，消除 v1.0.2 旧版"绝对 MCP 优先"的过度泛化。

#### 场景矩阵（S1-S6）

| 场景 | 工作场景 | 推荐工具 | 必须性 |
|---|---|---|---|
| **S1** | 接入新代码库（首次理解项目结构） | `get_repo_map` → `get_file_outline` | **必须** |
| **S2** | 探索大文件结构（>500 行，功能未知） | `get_file_outline` + `search_symbols` | 建议 |
| **S3** | 修改已知位置（行号已知或符号已知） | `Read` (offset/limit) **或** `get_symbol_source` | 直接合规 |
| **S4** | 跨文件搜索引用（找调用方/被调用方） | `check_references` / `find_importers` | 建议 |
| **S5** | 跨文件关键字搜索（grep 类） | `search_text` / `search_ast` | 替代 Bash grep |
| **S6** | 失败/不可用时 fallback | `repomix.pack_codebase` | 必须 fallback |

**必须性定义**：
- **必须**：违反触发自检警告，AI 应主动解释为何未用
- **建议**：默认使用，AI 可基于效率判断 fallback
- **直接合规**：任何方式都 OK，不需解释

#### 工作流（修订版）

```
任务开始 → 判断场景
├─ S1 接入新代码库 → 必须 L0 (get_repo_map) → L1 (get_file_outline)
├─ S2 大文件结构   → 建议 L1 (get_file_outline) + L2 (search_symbols)
├─ S3 修改已知位置 → 直接 Read (offset/limit) 或 L2 (get_symbol_source)
├─ S4 跨文件搜索   → 建议 L2 (check_references / find_importers)
├─ S5 跨文件关键字 → 建议 L2 (search_text / search_ast)
└─ S6 MCP 失败     → 必须 fallback (repomix → Read)
```

> **变更说明（v6.11 · 2026-07-01）**：原 L0-L3 漏斗被作为绝对命令，导致 70% "修改已知位置"场景被误判违规（meta-analysis 数据）。本次重设计将漏斗嵌入 S1-S6 场景矩阵，**保留** S1/S2 探索场景的 MCP 优先原则，**承认** S3 直接 Read 的精准优势。依据：`docs/superpowers/specs/2026-07-01-mcp-redesign.md` §2.1。

### 1.3 MCP 三件套职责分工

| 工具 | 最佳场景 | 兜底场景 | 限制 |
|------|---------|---------|------|
| **jCodeMunch-MCP** | Python 代码（AST 解析） | 任意结构化文件 | 需先 index_folder；worktree 默认未索引 |
| **Repomix** | 任意代码库（含 Markdown） | jcodemunch 索引失败时 | 一次性输出，无增量 |
| **Headroom-ai** | 大段 Markdown / 大文件压缩 | 长会话持续压缩 | hash 检索，不变更内容 |

### 1.4 唯一例外（5 条）

| # | 例外场景 | 处理 |
|---|---------|------|
| 1 | MCP 工具全部不可用（报错/超时/未索引） | 记录原因后用 Read，但**仍要尝试 repomix 兜底** |
| 2 | 文件 < 50 行（小配置文件等） | 可直接 Read |
| 3 | 已通过 MCP 定位，需要精确读取 1-3 行 | Read with offset/limit |
| 4 | **执行类操作**（git/cp/rsync/worktree） | 用 Bash，**不算违规** |
| 5 | **JSON/YAML/TOML 小配置**（< 30 行） | Read 全文，**不算违规** |

### 1.5 违规判定（4 档 · v6.11 场景感知 · 2026-07-01）

**核心变更**：从"通用阈值"（连续 3 Read、单次 > 100 行）改为"按场景判定"。原 §1.5 的"连续 3 次 Read 即红线"误判 S3 场景，予以删除。

| 等级 | 违规行为（按场景） | 自愈方法 |
|------|-------------------|---------|
| 🔴 **红线** | **S1 接入新代码库**未用 `get_repo_map` 直接 Read | 立即重写会话，先用 MCP 探查 |
| 🟠 **严重** | **S5 跨文件关键字搜索**未用 `search_text` 直接 Bash grep | 改用 `search_text` / `search_ast`（或 Read with offset/limit） |
| 🟠 **严重** | worktree 中未先尝试 `index_folder` 直接 Read | 优先 `index_folder(identity_mode="git")`；失败 → repomix 兜底 |
| 🟡 **中等** | MCP 不可用时未尝试 `repomix.pack_codebase` 兜底 | 自动调 repomix 兜底 |
| 🟡 **中等** | 长会话（> 30 轮）未用 `headroom_compress` | 定期压缩大段内容 |
| ✅ **合规** | **S3 修改已知位置**直接 Read（offset/limit） | 不算违规 |
| ✅ **合规** | **S5 单文件 grep** 直接用 Bash | 不算违规（仅限单文件） |
| ✅ **合规** | 文件 < 50 行（沿用 §1.4 例外 2） | 不算违规 |

> **变更说明**：原"连续 3 次 Read 即红线"过度泛化，70% 的"修改已知位置"场景被误判。`✅ 合规` 档明示 S3 / S5 单文件 / 小文件场景直接 Read 不算违规。依据：spec §3.2。

### 1.7 worktree 特殊流程（v6.11 自动化 fallback · 2026-07-01）

> **变更说明**：原 v6.4 流程要求"创建 worktree 后**立即** index_folder"（硬性命令），但 worktree 经常被 jcodemunch `trusted_folders` 拒绝，导致首次失败 → 心理放弃 → 退到 Bash/Read。新版改为"**优先尝试 + 自动 fallback**"。

**优先级自动反转**：worktree 中默认仍应尝试 MCP（因失败成本低），但失败时**自动**降级 repomix，**不再**算违规。

```bash
# 1. 进入 worktree → 优先尝试 MCP 索引
mcp__jcodemunch__index_folder(
  path=".worktrees/<name>",
  identity_mode="git",        # 关键：让 jcodemunch 识别为同一 git repo
  follow_symlinks=true        # 跟随 git worktree 符号链接
)

# 2. 失败 → 自动 fallback 到 repomix（不报错、不算违规）
mcp__repomix__pack_codebase(directory=".worktrees/<name>")

# 3. 仍失败 → 才 fallback 到 Read with offset/limit（最后手段）

# 4. 验证可解析（仅 S1 接入新代码库场景必须）
mcp__jcodemunch__resolve_repo(path=".worktrees/<name>")
```

**关键差异（vs v6.4）**：
- ✅ "立即索引" → "优先尝试"（失败不阻断）
- ✅ `resolve_repo` 步骤 2 → 步骤 4（仅 S1 场景必须）
- ✅ "失败兜底" → "自动 fallback"（语义降级）
- 🆕 失败时由 repomix 接续，不需 AI 手动判断

### 1.10 🆕 测试纪律（v1.3.2 加入 · 教训来自 L#002 specs 卡死事故）

> **背景**：2026-07-03 specs 安装卡死事故。AI 助手在 5 轮本地测试中**每次都加 `--skip-specs` 跳过**真正的问题，导致"端到端验证通过"但用户实际命令卡死 1 powershell + 7 个 git 子进程，**永不退出**。详细事故链见 `docs/lessons-learned.md` L#002。
>
> **核心混淆**：`git pull` 在远端不存在时**不是 fail** 而是 **hang**（反复重试 TCP/认证）。`if ! cmd` 抓不到，`2>/dev/null` 吞不掉。必须 `timeout N` 主动设上限。

**5 条硬规则**（任何 release/fix PR 涉及 install 或 deploy 改动前必查）：

1. **黄金路径测试纪律**：每个 release/PR 必跑**用户最小命令**（零 flag）端到端至少 1 次。开发期 `--force` / `--skip-specs` / `--only` 等是便利**不是替代**。
2. **区分 fail vs hang**：
   - **失败（sync fail）**：返回非 0 → `if ! cmd` 能捕获
   - **挂起（hang）**：永不返回 → 必须 `timeout N cmd` 主动设上限
   - `2>/dev/null` / `2>&1 | Out-Null` **只吞错误信息，不终止挂起**
3. **测试时禁用开发期 flag**：若加 flag 才"测试通过"，立刻怀疑"用户裸跑会怎样"。"测试通过 ≠ 真实可用"。
4. **脏状态 = 测试失败**：本地有遗留 `~/.xxx/` 脏目录（指向不存在 URL）时跑 deploy 100% 触发 hang。**测试前清环境**应成为肌肉记忆。
5. **加新功能时 red team 自己的设计**：
   - 依赖仓库不存在？远端返回 5xx？网络慢到 timeout？用户在防火墙后？
   - **任何一个没设计降级路径的依赖都是定时炸弹**
   - 例：5 轮前 specs 外部化功能，**5 轮没人问"如果 specs 仓库不存在会怎样"**

**自查清单**（每次写 install/deploy 脚本前）：

- [ ] 是否引入了新的外部依赖（git clone、API、远程服务）？
- [ ] 外部依赖不可达时，脚本是 fail（可控）还是 hang（用户卡死）？
- [ ] 是否给了"如果依赖不存在怎么办"的降级路径（timeout / 默认跳过 / 完全删除）？
- [ ] 本地测试是否至少跑过 1 次"用户裸命令"（无任何 flag）？
- [ ] 本地测试前是否清掉脏目录（`rm -rf` 模拟首次安装）？

## 🔴 2. 事实优先硬规则（分析端 · 本项目最高优先级 · 不可违反 · 2026-06-29 加入）

> **针对 2026-06-29 v5.4 兼容性胡乱分析事故**（AI 基于错误假设套用通用话术被用户当场识破），
> 本项目强制执行以下 5 条事实优先规范：

1. **项目分析前必查 5 项事实**（参见 `skills/evidence-first/SKILL.md`）：
   - `git log --oneline -10`（最近 10 次提交）
   - `README.md` / `AGENTS.md`（项目自述）
   - `pyproject.toml` / `package.json`（版本声明）
   - `docs/` 下最近的设计文档 / CHANGELOG
   - `git log --since="3 months ago" --oneline | wc -l`（3 个月活跃度）

2. **每句话必须标注 [F]/[H]/[P]**：
   - `[F]` 事实（已通过 git/file/docs 验证，可追溯）
   - `[H]` 假设（基于经验但未验证，需明确标注）
   - `[P]` 原则（通用工程参考，优先级最低）

3. **不确定 = 说"我不清楚"**：禁止凑答案 / 编造 / 套模板

4. **长篇论述前自检 4 问**：
   - 我有 [F] 事实依据吗？
   - [H] 假设明确标注了吗？
   - 错了损失大吗？（高损失必须 [F] 主导）
   - 能说"我不清楚"吗？

5. **判断必须可追溯到事实**：每条结论附"事实依据 + 来源 + 验证方式"

**未完成事实清单 = 禁止进入分析论述**。

**违规判定**：长篇论述（> 5 段）无 [F] 标注 = 视为红线违规，自动加载 `evidence-first` 技能重写。

**事故教训库**：`docs/lessons-learned.md`（含本事故的完整记录 + 修复措施）

## 🔴 3. 用户交互红线（沟通端 · 全局适用 · 单点真源 · v6.10 升级：5 条 + §3.6 自查清单）

> **任何涉及用户决策、方案选择、确认的交互，必须遵循以下 5 项硬要求。**

1. **必须使用 `AskUserQuestion` 工具** — 列出 2-4 个选项（含推荐），让用户通过选择框选择，**不允许要求用户输入自由文本**
2. **推荐项必须标注 `(推荐)` 并说明理由** — 第一个选项为推荐项，描述中说明为什么推荐
3. **不推荐项必须说明理由** — 每个非推荐选项的描述中说明不推荐或需谨慎的理由
4. **禁止开放式追问** — 不允许"你觉得呢？""还需要什么？""你还想做吗？"等需要用户组织语言回答的提问
5. **🆕 决策点必须用 AskUserQuestion 触发**（2026-07-01 加入）— 任何"## 🎯 决策点 / 决策 X / 是否要..."的输出，**必须**配对 `AskUserQuestion` 工具调用，**禁止**用 markdown 文字列决策点。**这是硬约束**：决策点文字 + 不调 AskUserQuestion = 红线违规。

**违反以上任一条 = 视为阻塞 Bug，必须自愈修复。**

#### 3.7.1 §3.6 自查清单第 6 项（v6.11 新增）

#### 3.7.2 违规判定

- 建议段落含 ≥ 2 个候选但未调 AskUserQuestion = 🔴 红线违规（与"决策点"违规同级）
- 单一推荐（如 `**建议 X，因为 Y**`）= 合规（无选项列表，不触发 AskUserQuestion）
- 纯事实陈述 / 引用 / 清单（无选项性内容）= 合规

#### 3.7.3 6 条避免方法（v6.11 升级）

1. 写 "## 🎯 决策点" 之前先自问 → **立即调** AskUserQuestion（不要先写 markdown 再调工具）
2. 红线 3.6 五项 checklist 每次输出结束前必走
3. "如需..."型句子默认触发 AskUserQuestion
4. 区分"建议"与"决策"的写法：
   - **建议 + 单一推荐**（合规）：`**建议 X，因为 Y**`
   - **建议 + 多候选**（违规 → 改用 AskUserQuestion）：禁止 markdown 列表
   - **决策 + 多选项**（必须 AskUserQuestion）
5. 养成"先调 AskUserQuestion 再写 markdown 选项"的肌肉记忆
6. **🆕 自检触发条件**：任何输出末尾含"如 / 或 / 可以 / 推荐 / 建议 / 候选" + 选项列表 → 必走 AskUserQuestion

## 🔴 4. 摘要输出红线（产出端 · 全局适用 · 单点真源 · 2026-06-30 加入 · v6.10 升级决策点硬约束）

### 4.1 触发场景（必须输出摘要）

## 🔴 5. 完成前验证红线（诚信端 · 全局适用 · 单点真源 · 2026-06-30 加入 · v6.9 升级）

> **针对大模型"幻觉式完成"问题**（AI 在未真正执行任务时宣称已完成，违反诚信），
> 本项目强制执行"证据先行"铁律：**无验证证据 = 无完成声明**。

### 5.1 铁律（Iron Law）

```
NO COMPLETION CLAIMS WITHOUT FRESH VERIFICATION EVIDENCE
```

"完成"的三要素（**三者缺一 = 不得宣称完成**）：

1. **任务已实际执行** —— 不是"准备执行"、不是"应该执行了"
2. **结果已实际输出** —— 不是"理论上能输出"、不是"应该输出了"
3. **验证命令已运行** —— 在**本轮消息**中跑过命令、读过输出

### 5.2 触发场景（10 类常见执行任务 · 必须验证后才能宣称完成）

| # | 任务类型 | 验证证据（必须） |
|---|---------|---------------|
| 1 | 项目调研 | 调研报告已写入文件 / 已在输出中呈现（含具体发现） |
| 2 | 网络搜索 | `WebFetch` / `WebSearch` 已返回实际结果（不是"调用了"工具） |
| 3 | 代码探查 | MCP 工具已返回实际代码内容 / 符号源码（不是"探查了"） |
| 4 | 代码分析 | 分析结论已附带具体行号 / 文件路径 / 符号引用 |
| 5 | 结构设计 | 设计文档已写入文件 / 文档大纲已呈现 |
| 6 | 功能实现 | 代码已实际写入文件（`git diff` 可查） |
| 7 | 代码编写 | 文件已创建 / 已修改（`ls` 或 `git status` 可证） |
| 8 | 测试运行 | 测试命令已执行 + 输出已读取 + 失败计数已确认 |
| 9 | Bug 修复 | 原始症状复现测试 + 修复后测试已通过（**红绿循环**验证） |
| 10 | 任何"已完成"声明 | 验证命令已在**本轮消息**运行 + 输出已读取 |

### 5.3 豁免场景（无需验证）

- 闲聊 / 简单问答（"你好"、"版本号多少"）
- 用户明确表示"不要求验证 / 跳过验证"
- 纯建议 / 方案推荐（不涉及"已完成"声明）
- 与摘要红线相同的豁免：单行错误信息 / 执行类纯状态汇报 / 用户明确要求"不要摘要"

### 5.4 违规判定（6 类典型违规）

| 违规行为 | 自愈方法 |
|---------|---------|
| 宣称"已完成 X"但 X 的验证命令未在本轮运行 | 红线违规 → 立即重跑验证 + 客观陈述结果 |
| 工具调用失败 / 超时 / 报错，仍宣称"已完成" | 红线违规 → 客观陈述失败状态，**不得**宣称完成 |
| 输出被截断 / 部分完成，仍宣称"已全部完成" | 红线违规 → 标注未完成部分 |
| 突发不可逆状态（会话中断 / 上下文丢失 / 工具崩溃），仍宣称"已完成" | 红线违规 → **只**客观陈述不能完成的现状，**不得**宣称已完成 |
| 使用"应该完成"、"差不多"、"已经做了"等模糊措辞 | 视为违规 → 必须给出具体证据（行号 / 退出码 / 失败计数） |
| 转述"agent / 工具报告成功"而未独立验证 | 视为违规 → 必须独立验证（VCS diff / 输出重读） |

### 5.5 自查清单（会话结束前必查）

- [ ] 本轮消息中所有"已完成"声明是否都有对应的验证命令运行？
- [ ] 验证命令的输出是否已**实际读取**（不是"看起来 OK"）？
- [ ] 工具调用失败时是否已客观陈述失败状态（未宣称完成）？
- [ ] 输出截断时是否已标注"未完成"部分？

> **来源**：单一真源 = 本章节。**详细方法论**（铁律 + Gate Function + 常见失败表 + 借口防范表）见 `skills/verification-before-completion/SKILL.md`；本章是**红线版**（必须遵守的硬约束）。两者**并存互引** —— 红线指向技能的方法论，技能指向红线的硬约束。

## 🔴 6. 进度汇报红线（运行端 · 单点真源 · 2026-07-01 加入）🆕

> **针对长任务"假死感"痛点**（用户不知道 AI 在干什么、是否还活着、何时完成），
> 本项目强制执行"运行端节奏控制"铁律：**预估 ≥ 10 步 或 > 2 分钟的任务，必须分段汇报进度**。

### 6.1 铁律

预估执行步骤 ≥ 10、调用 subagent、或总时长预期 > 2 分钟的任务，必须在**每 5-10 步或每主要阶段完成时**输出结构化进度，禁止 > 30 分钟静默。

### 6.2 触发场景（必须汇报）

- 多步骤执行（≥ 10 步工具调用）
- 多文件改动（≥ 5 个文件）
- 跨 skill 编排（orch 调度 ≥ 2 个 skill）
- 调用 subagent（dispatching-parallel-agents / subagent-driven-development）

### 6.3 豁免场景（无需汇报）

- 单步操作（1-2 工具调用）
- 短任务（< 5 步 / < 1 分钟）
- 用户明确表示"不需要进度，直接做完"

### 6.4 进度结构（固定模板）

```markdown
## 📊 进度汇报 (N/M)
- **已完成**：[步骤列表，简短]
- **当前阶段**：[具体阶段名]
- **下一步**：[即将做什么]
- **阻塞**：[如有，未阻塞则填"无"]
- **预计剩余**：[X 步 / Y 分钟]
```

### 6.5 与"摘要输出红线"的协调

- **进度汇报**：**中途**节奏控制（每 5-10 步）
- **摘要输出**：**任务结束**结构化摘要
- 两者不冲突：进度汇报简短，结束时归入完整摘要

### 6.6 违规判定

- 满足触发条件但 > 30 分钟无任何状态输出 → 红线违规
- 仅转述"还在做"、"差不多了"等模糊措辞 → 红线违规 → 必须用固定模板结构
- 与完成前验证红线冲突时优先级：**完成前验证（诚信端） > 进度汇报（运行端）**

## 🔴 8. 一致性核对红线（追踪端 · 单点真源 · 2026-07-03 加入）🆕

> **针对复杂任务"做完几个忘几个"痛点**（已实现与文档脱节、未实现功能被遗忘、
> 模块流程未打通、潜在问题累积），本项目强制执行"多任务累积一致性核对"铁律：
> **每完成 3 个子任务或 1 个跨模块改动后，必须做需求↔实现↔文档一致性核对，
> 不得连续累积超过 3 个任务不回扫。**

### 8.1 铁律

满足任一触发条件（§8.2）时，必须停下执行，按用户选定维度做一致性核对（§8.3），
输出差异报告（§8.4），由用户通过 AskUserQuestion 决定不一致项的处置（§8.5）。

> **方法论语源**：本红线定义"何时查 + 输出什么 + 查完怎么办"。
> 四维度的检查清单不在此重复，引用 `system-review` 技能（遵循单点真源原则）。

### 8.2 触发条件（满足任一即触发 · 计数器必须维护）

| # | 条件 | 说明 |
|---|------|------|
| 1 | 累计完成 **3 个子任务** | 自上次核对后计数，核对后清零 |
| 2 | 完成 **1 个跨模块改动** | 改动触及 ≥ 2 个模块边界（接口/数据流/契约） |
| 3 | 完成 **1 个架构级改动** | 新增/删除模块、改分层依赖方向、改核心数据模型 |
| 4 | 用户显式要求 | "做个一致性核对" / "查一下偏离" |

> **计数器实现**：本红线只规定"必须维护计数"，不规定实现方式（会话状态 / `.loop-state-*.json` /
> `.orchestrate-state.json` / 上下文记忆 均可）。**长会话上下文丢失 ≠ 豁免核对**——
> 不确定时优先触发，宁可多查不可漏查。

### 8.3 核对维度（触发后用 AskUserQuestion 让用户选维度）

触发核对时，**必须先用 AskUserQuestion 让用户选本次核对维度**（multiSelect: true）：

| # | 维度 | 检查内容 | 方法论语源 |
|---|------|---------|-----------|
| 1 | **需求↔实现** | 最初计划/PRD 清单 vs 已实现代码，逐条打勾；发现 [GAP]（需求有代码无）或 [EXTRA]（代码有需求无） | 本红线自含 |
| 2 | **模块间横向一致性** | 接口契约/字段命名/错误约定/状态机 在各模块间是否一致 | `system-review` Step 1 |
| 3 | **端到端链路打通** | 选 2-3 个核心用例，从入口追到出口，找断点 | `system-review` Step 2 链路完整 |
| 4 | **文档↔代码一致性** | README / API 文档 / CHANGELOG 是否与当前代码状态一致 | `system-review` Step 3 过时文档 |

> **默认推荐**：首次 AskUserQuestion 选项含全部 4 维度（标注 `(推荐)`）+ 轻量版（仅 1+3）+
> 单维度查。用户可选 ≤ 4 个维度，**至少必选 1 个**。
> 方法论不在本红线重复——维度 2/3/4 的检查清单引 `system-review` 技能。

### 8.4 输出格式（固定模板 · 核对完必输出）

```markdown
## 🔍 一致性核对报告（第 N 次 · 触发原因：[3任务累积/跨模块/架构级/用户要求]）
- 任务计数器：本次核对前 = X 任务，核对后清零

### 维度1: 需求↔实现
- [GAP]    需求 R3 "用户注销" 未见实现代码
- [EXTRA]  代码 src/logout.py 未在原始需求中
- [OK]     需求 R1/R2/R4-R10 均已实现

### 维度2: 模块间横向一致性（引 system-review Step 1）
- [CRITICAL] API 层 User.id 类型 string vs DB 层 int
- [HIDDEN]   订单状态"已取消" 命名跨模块不一致

### 维度3: 端到端链路（引 system-review Step 2 链路完整）
- 用例"下单"：✅ 全链路打通
- 用例"取消订单"：❌ OrderService → 库存回滚 断点

### 维度4: 文档↔代码一致性（引 system-review Step 3 过时文档）
- [STALE] README §3 仍写"使用 SQLite"，实际已迁 Postgres
```

### 8.5 处置协议（每个不一致项或分组必走 AskUserQuestion）

发现不一致 → **不得**自行决定修复/忽略，必须为每个差异项（或合理分组）调 AskUserQuestion：

| 选项 | 含义 | 推荐场景 |
|------|------|---------|
| 立即修复 | 阻断下个任务，先修这个 | CRITICAL / 阻塞型（默认 `(推荐)`） |
| 登记后续修复 | 记入 TODO，继续下个任务 | HIDDEN / 非阻塞型 |
| 标记已知边界 | 不修但文档说明 | 设计取舍 / 暂不支持 |
| 忽略 | 明确不修也不记录 | 误报 / 无意义 |

> 与红线 3（用户交互）协同：处置本身就是决策点，必须走 AskUserQuestion，禁止文字列选项。

### 8.6 豁免场景（无需核对）

- 单步任务（1-2 个子任务就完成，未到阈值）
- 用户明确"跳过核对 / 不要核对"
- 纯调研 / 分析任务（无代码改动）
- 单文件修复（不跨模块、不动架构）

### 8.7 违规判定

| 违规行为 | 自愈方法 |
|---------|---------|
| 满足触发条件未触发核对 | 红线违规 → 立即补做核对 |
| 触发后未用 AskUserQuestion 让用户选维度 | 红线违规 → 补走维度选择决策 |
| 仅核对部分维度但未经用户确认 | 红线违规 → 补齐缺失维度或显式经用户确认豁免 |
| 发现不一致未走 AskUserQuestion 处置 | 红线违规 → 补走决策流程（与红线 3 同级） |
| 核对后未输出 §8.4 报告 | 视为冗余 → 重写追加报告 |

### 8.8 与其他红线的关系

| 协同红线 | 关系 |
|---------|------|
| 🔴 5 完成前验证 | 红线 5 = 单任务粒度"完成前"；红线 8 = 多任务累积"中途回扫"。**互补不冲突** |
| 🔴 3 用户交互 | 红线 8 §8.3 维度选择 + §8.5 处置都必须走 AskUserQuestion，是红线 3 的具体应用 |
| 🔴 4 摘要输出 | 红线 8 报告本身遵循红线 4 结构化输出模板 |

## 🔴 9. 工程实践红线（开发端 · 单点真源 · 2026-07-03 新增 · v1.0.4）

> **针对方案选型 / 技术选型 / 修复 / 重构 / 小步快跑 / 决策记录 6 维度共 16 条硬约束**
> **触发场景**：任何 AI 写代码 / 推荐方案 / 技术选型 / Bug 修复 / 重构 / 提交决策时触发
> **论源原则**：每条规则末尾标注方法论来源 skill（单点真源，不重复检查清单）
> **强度等级**：分 3 档（🔴 红线 / 🟠 严重 / 🟡 中等），与 §1.5 协同

### 9.1 方案选型（4 条）— 论源：brainstorming / clean-code / software-architecture

- **R1.1 反选项清单** [🔴] — 任何方案推荐前必须先列 ≥ 2 个"被否决备选"+ 否决理由；禁止"只推荐 A 不说为什么不选 B"
- **R1.2 机会成本可见** [🟠] — 推荐方案必须说明"放弃了什么换取什么"（trade-off 透明）
- **R1.3 YAGNI** [🔴] — 禁止为"未来可能用到"提前抽象 / 预留接口 / 加配置开关
- **R1.4 可回滚红线** [🟠] — 禁止 all-in 不可逆选择（DB schema 无回滚 / 协议硬编码无版本）

### 9.2 技术选型（1 条）— 论源：software-architecture

- **R2.4 PoC / Spike 时间盒** [🔴] — 重大选型必须 spike（≤ 4h），超时强制决策；禁止"边想边写大段代码"

### 9.3 修复原则（2 条）— 论源：systematic-debugging / code-reviewer

- **R3.1 根因分析** [🔴] — 禁止症状级修复（吞异常 / 改判断绕过 / 加默认值掩盖）；必须先找根因再修
- **R3.3 修复 scope = bug scope** [🟠] — 禁止"顺手重构"+ unrelated cleanup 进 fix commit（PR 纯净原则）

### 9.4 重构原则（2 条）— 论源：refactoring

- **R4.1 测试保护** [🔴] — 无测试覆盖代码禁止重构（先补测试，再重构）
- **R4.4 机械重构优先** [🟠] — rename / extract function / move class 优先于架构重构（先小后大）

### 9.5 小步快跑 / 简洁高效（6 条 · 核心）⭐ — 论源：clean-code / brainstorming / code-reviewer / evidence-first

- **R5.1 KISS 红线** [🔴] — 禁止为"优雅 / 通用 / 扩展性"增加不必要的抽象层 / 装饰器 / 元编程
- **R5.2 减法红线** [🔴] — 新增 ≥ 5 行代码前必须先回答"能不能不写？能不能复用现有？"
- **R5.3 Tracer Bullet 红线** [🔴] — 新功能必须先打通端到端骨架（walking skeleton），再补血肉；禁止"完整设计 + 完整实现"一气呵成
- **R5.4 PR Size 红线** [🔴] — 单 PR 改动 ≤ 200-400 行（不含自动生成 / lock 文件）
- **R5.6 Working > Comprehensive** [🟠] — 禁止"完整文档先行"；先让代码 work，再补文档
- **R5.7 Spike & Iterate** [🟠] — 不确定时先写 spike（探索代码）→ 再写实现；禁止"边想边写大段代码"

### 9.6 决策记录（1 条）— 论源：software-architecture

- **R6.2 ADR 红线** [🟠] — 重大决策必须写 ADR（含上下文 / 选项 / 决定 / 后果），存于 `docs/superpowers/adr/YYYY-MM-DD-<topic>.md`

### 9.7 违规判定（3 档 · 与 §1.5 协同）

| 等级 | 违规行为 | 自愈方法 |
|------|---------|---------|
| 🔴 **红线** | 违反 R1.3 / R2.4 / R3.1 / R4.1 / R5.1 / R5.2 / R5.3 / R5.4 | 立即重写，先解释为何未遵守 |
| 🟠 **严重** | 违反 R1.1 / R1.2 / R1.4 / R3.3 / R4.4 / R5.6 / R5.7 / R6.2 | 必须给出 explanation，或放弃该方案 |
| 🟡 **中等** | 论源引用缺失 / 与现有 skill 方法论冲突 | 自查清单补充说明，登记 TODO |

### 9.8 自查清单（会话结束前必查 · 8 项）

- [ ] **R1** 方案推荐是否含 ≥ 2 个反选项 + 机会成本说明？
- [ ] **R2** 新依赖 / 框架引入是否经过 spike 验证（≤ 4h）？
- [ ] **R3** Bug 修复是否给出根因（不是症状级）？fix commit 是否纯净？
- [ ] **R4** 重构前是否有测试覆盖？是否独立 commit？
- [ ] **R5** 代码是否经过"能不能不写 / 复用现有"检查？是否先 Tracer Bullet？
- [ ] **R5** PR 是否 ≤ 400 行？是否有不必要的抽象？
- [ ] **R6** 重大决策是否写了 ADR？
- [ ] **论源** 新增红线 / 修改是否引用了对应 skill 作为方法论？
